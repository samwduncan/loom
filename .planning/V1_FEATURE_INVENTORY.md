## 2. Streaming Pipeline

### 2.1 Architecture

```
WebSocket message → useChatRealtimeHandlers → 60ms buffer → setChatMessages → TurnBlock → Markdown
```

| Component | Role | Files |
|-----------|------|-------|
| Stream buffer | 60ms accumulation window, flush via `requestAnimationFrame` | `useChatRealtimeHandlers.ts` (lines ~200-260) |
| Content block assembly | Handles `content_block_start`, `content_block_delta`, `content_block_stop` for Claude | `useChatRealtimeHandlers.ts` |
| Codex item handling | Processes `agent_message`, `reasoning`, `command_execution`, `file_change`, `mcp_tool_call` items | `useChatRealtimeHandlers.ts` |
| Gemini handling | Simpler: `gemini-response` with text content | `useChatRealtimeHandlers.ts` |
| isStreaming flag | Set per-message, propagated to Turn, controls aurora/shimmer | `useTurnGrouping.ts`, `TurnBlock.tsx` |
| Force-streaming | Last turn marked streaming when `isLoading=true` even without streaming messages | `useTurnGrouping.ts` (line 112-116) |

### 2.2 Provider-Specific Message Types

**Claude:**
- `claude-response` — streaming chunks (content_block_delta for text, input_json_delta for tools, thinking delta for thinking)
- `claude-output` — final complete message
- `claude-complete` — session done
- `claude-error` — error during processing
- `claude-permission-request` / `claude-permission-cancelled`
- `claude-status` — status text updates
- `claude-interactive-prompt` — interactive questions

**Codex:**
- `codex-response` — item-based (agent_message, reasoning, command_execution, file_change, mcp_tool_call, error)
- `codex-complete` / `codex-error`

**Gemini:**
- `gemini-response` — text content
- `gemini-error`
- `gemini-tool-use` / `gemini-tool-result`

### 2.3 Streaming Visual States

| State | Visual | Files |
|-------|--------|-------|
| Pre-token (thinking) | Full aurora glow, centered "Thinking..." label with elapsed/tokens | `AuroraOverlay.tsx` |
| Active streaming | Aurora at reduced opacity, thinking label top-left | `AuroraOverlay.tsx` |
| Completing | Aurora fades out (500ms), content gains full opacity | `AuroraOverlay.tsx` |
| Activity status | Semantic text: "Reading file.ts...", "Running command...", etc. | `useActivityStatus.ts`, `StatusLine.tsx` |

**Status:** WORKS — all three providers stream correctly.

**Complexity:** Complex — the ~1050-line `useChatRealtimeHandlers.ts` is the most complex single file. Multi-provider support adds significant branching.

## Complexity Summary for V2 Reimplementation

| Category | Complexity | Notes |
|----------|-----------|-------|
| Chat/Messaging | **Complex** | Turn grouping, scroll management, streaming markdown are tightly coupled |
| Streaming Pipeline | **Complex** | ~1050-line handler, multi-provider branching, 60ms buffering |
| Tool Calls | **Complex** | Config-driven system with many tool types, diff viewer, subagent containers |
| Thinking/Reasoning | **Medium** | Self-contained, clear boundaries |
| Session Management | **Complex** | Race conditions, multi-provider, paginated loading, URL routing |
| Input Area | **Complex** | ~500-line state hook, mentions, commands, images, permissions |
| Settings | **Medium** | Standard CRUD, but multi-provider auth is tricky |
| Terminal/Shell | **Medium** | xterm.js handles heavy lifting |
| Connection Management | **Medium** | Single connection, auto-reconnect, extensive message types |
| Theme/Styling | **Medium** | Token system deeply integrated |
| Navigation | **Simple-Medium** | Standard routing + sidebar |
| Error Handling | **Simple-Medium** | Distributed, not centralized |

**Overall:** The frontend is a significant engineering effort. The most complex subsystems are the streaming pipeline, turn grouping, and tool call rendering. A V2 rewrite should prioritize getting these right early, as they are the core differentiators of the Loom experience.
