## S-Grade Issues

### [S] drawerOpen event does not exist in @react-navigation/drawer v7
**Source:** Guard + Hunter + Architect (3/3) | **Confidence:** High
**Plan:** 76.1-03 Task 1 step 5
**Description:** Plan uses `screenListeners={{ drawerOpen: () => ... }}` but `@react-navigation/drawer` v7.3.9 only exposes: `drawerItemPress`, `transitionStart`, `transitionEnd`, `gestureStart`, `gestureEnd`, `gestureCancel`. The `drawerOpen` event does not exist. The fallback `state` event with `history` inspection uses undocumented internals and `any` typing.
**Fix:** Use `transitionEnd` event: `transitionEnd: (e) => { if (!e.data.closing) haptic.transition(); }`. Or use `navigation.addListener('transitionEnd', ...)` in DrawerContent which already has `props.navigation`.

### [S] Plan 02 FlatList padding reasoning is confusing + missing useSafeAreaInsets import
**Source:** Guard + Hunter + Architect (3/3) | **Confidence:** High
**Plan:** 76.1-02 Task 1
**Description:** The plan's "Wait -- inverted FlatList reverses the meaning" self-correction mid-instruction will confuse executors. The final answer (paddingBottom = visual top) is correct but the reasoning is contradictory. Additionally, `useSafeAreaInsets` must be imported in ChatScreen for `HEADER_HEIGHT = insets.top + 56`, but this import is not in read_first or acceptance criteria. If missed, `insets` is undefined and `HEADER_HEIGHT` becomes NaN.
**Fix:** Remove self-correction, state clearly: "In inverted FlatList, paddingBottom = visual top." Add `useSafeAreaInsets` import to acceptance criteria. Add fallback: `(insets?.top ?? 44) + 56`.

### [S] Plan 02 model indicator data source unspecified
**Source:** Guard (B) + Hunter (B) + Architect (S) — all three found it | **Confidence:** High
**Plan:** 76.1-02 Task 1
**Description:** D-12 model indicator says "come from stream store or be passed as a prop. Read ChatHeader.tsx to determine how" — this is a vague "figure it out" instruction that violates deep-work rules. ChatHeader is currently prop-driven (title is a string prop). The plan doesn't specify the store selector, import path, or null fallback.
**Fix:** Specify: Add `modelName?: string` prop to ChatHeaderProps. In ChatScreen: `const modelName = useStreamStore((s) => s.modelName)`. Pass `modelName={modelName}`. Display `modelName ?? ''`. Keep ChatHeader prop-driven.

### [A] Typing-begins haptic fatigue on rapid clear-retype
**Source:** Hunter + Architect | **Confidence:** Medium
**Plan:** 76.1-03 Task 1
**Description:** useEffect on `text` fires every keystroke. haptic.transition() fires each time user clears all text and starts typing again. Rapid editing produces repeated Medium haptics violating D-14 "musical haptics, not blanket buzzing."
**Fix:** Add 2-second cooldown via `lastTypingHapticRef` before haptic can fire again.

### [A] UserBubble shadow applied via inline style, not createStyles
**Source:** Hunter | **Confidence:** High
**Plan:** 76.1-01 Task 2
**Description:** Plan says "add rimLight spread after existing shadow spread: `...t.shadows.subtle, ...t.rimLight`" in createStyles. But actual UserBubble applies shadow via inline style array `[styles.bubble, theme.shadows.subtle]`, not in createStyles. Following plan literally duplicates the shadow.
**Fix:** Instruct: "Add `...t.rimLight` to bubble style in createStyles. Do NOT add `...t.shadows.subtle` — shadows are already applied inline."
