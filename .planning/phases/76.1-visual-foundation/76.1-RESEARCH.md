# Phase 76.1: Visual Foundation - Research

**Researched:** 2026-04-04
**Domain:** iOS visual polish -- surface hierarchy, glass surfaces (expo-blur), interaction weight (haptics + springs), reference app density matching
**Confidence:** HIGH

## Summary

Phase 76.1 is a visual polish phase that refines 8 existing components (not greenfield) to match the quality bar of ChatGPT iOS and Claude iOS. The foundation is already solid: the theme system is complete with typed tokens (4-tier surfaces, 6 named springs, shadows, radii, typography), createStyles pattern is established, and most components already have spring animations and haptic feedback. The work is primarily auditing, adjusting, and layering -- not building from scratch.

The two main technical additions are: (1) adding expo-blur BlurView to ChatHeader and Composer for glass surfaces, and (2) creating a centralized haptics utility that encodes the D-14 haptic hierarchy. Everything else is refinement: verifying surface colors match the device-optimized values in `colors.ts`, adding rim light borders (D-03), auditing every interactive element for spring + haptic coverage, and tuning density to match reference apps.

**Primary recommendation:** Start with the two hardest, riskiest items -- Composer glass (D-07 keyboard blur collision) and ChatHeader glass (D-04) -- to get early feedback on whether glass works or falls back to opaque. Then sweep remaining surfaces (drawer density, chat density, haptic hierarchy, spring audit) as lower-risk tuning work.

### Pattern 1: Glass Surface (BlurView Wrapper)

**What:** Wrap a component in expo-blur BlurView with dark overlay for glass treatment.
**When to use:** Nav header and composer (floating over scrolling content).
**Critical rule from expo-blur docs:** BlurView must be rendered AFTER dynamic content (e.g., FlatList). If rendered before, blur effect won't update.

```typescript
// Source: expo-blur official docs + existing SessionSearch.tsx pattern
import { BlurView } from 'expo-blur';

// Glass container pattern (already established in SessionSearch, ConnectionBanner)
<View style={[styles.container, { overflow: 'hidden', borderRadius: 16 }]}>
  <BlurView intensity={40} tint="dark" style={styles.blur}>
    <View style={styles.overlay}>
      {/* Content */}
    </View>
  </BlurView>
</View>

// overlay style: backgroundColor: 'rgba(0,0,0,0.35)'
// border: borderWidth 1, borderColor: theme.colors.border.subtle
```

### Pattern 2: Haptic Hierarchy Utility

**What:** Centralized module mapping interaction types to correct haptic feedback.
**When to use:** Every interactive element calls the semantic function, not raw expo-haptics.

```typescript
// mobile/lib/haptics.ts
import * as Haptics from 'expo-haptics';

export const haptic = {
  /** Button press, selection, scroll-to-bottom tap */
  tap: () => Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Light),
  /** Drawer open/close snap, state transitions, session switch */
  transition: () => Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Medium),
  /** Tool call complete, session complete */
  success: () => Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success),
  /** Permission request appears */
  warning: () => Haptics.notificationAsync(Haptics.NotificationFeedbackType.Warning),
  /** Error state transition */
  error: () => Haptics.notificationAsync(Haptics.NotificationFeedbackType.Error),
  /** Swipe threshold crossed, list item long-press, selection change */
  selection: () => Haptics.selectionAsync(),
} as const;
```

### Pattern 3: Rim Light Border (D-03)

**What:** Subtle top-edge highlight on raised surfaces for depth perception.
**When to use:** All `surface-raised` containers (UserBubble, ToolChip, cards, composer).

// Add to theme.ts

```

```

### Anti-Patterns to Avoid

- **Blanket haptics:** Don't add `Haptics.impactAsync()` to everything. Use the semantic hierarchy (D-14). Musical haptics, not buzzing.
- **AnimatedPressable everywhere:** Not every View needs press animation. Only interactive elements (Pressable, TouchableOpacity) get Micro spring.
- **Multiple BlurViews stacking:** On screen simultaneously (header glass + composer glass + connection banner glass) can compound GPU cost. Monitor frame rate.
- **BlurView before FlatList:** expo-blur docs explicitly warn that rendering BlurView before dynamic content causes stale blur. The header BlurView must be rendered after MessageList or use zIndex/absolute positioning.
- **Linear/ease animations:** The Soul doc prohibits linear/ease for motion (position, scale, opacity on interactive elements). Only color transitions use withTiming.
- **Overriding iOS system behaviors:** Never fight status bar, Dynamic Island, home indicator, edge-swipe back, or keyboard animation.

## Common Pitfalls

### Pitfall 1: BlurView Render Order
**What goes wrong:** Glass header appears as solid dark rectangle (blur doesn't update).
**Why it happens:** expo-blur docs: "The blur effect does not update when BlurView is rendered before dynamic content (e.g., FlatList)."
**How to avoid:** Render the ChatHeader AFTER MessageList in the component tree, or use absolute positioning with zIndex to layer it above. The current ChatScreen renders `<ChatHeader>` before `<MessageList>` -- this order must be reversed in the DOM while maintaining visual stacking via position absolute.
**Warning signs:** Header appears as opaque dark surface instead of transparent blur.

### Pitfall 2: Keyboard-Blur Collision (D-07)
**What goes wrong:** Composer BlurView lags 1-2 frames behind keyboard animation, creating a visual "tear" between composer and keyboard.
**Why it happens:** BlurView re-compositing can't keep up with 60fps keyboard animation. The blur needs to recalculate every frame as the composer moves.
**How to avoid:** Prototype early. If stutter occurs, fall back to opaque `surface-raised` with `borderTopWidth: 1, borderTopColor: border-subtle`. The fallback must look intentional.
**Warning signs:** Visible gap or flash between keyboard top edge and composer bottom edge during keyboard show/hide.

### Pitfall 3: Stacked BlurViews GPU Cost
**What goes wrong:** FPS drops below 55 during streaming with header glass + composer glass both active.
**Why it happens:** Each BlurView is a separate compositing layer. Two blur layers + streaming FlatList re-renders + Reanimated animations = GPU pressure.
**How to avoid:** Test with both blur layers active while streaming a long response. If FPS drops, remove composer glass first (fallback to opaque) -- header glass is more visually impactful.
**Warning signs:** Janky scrolling during streaming, frame drops visible in React Native Perf Monitor.

### Pitfall 4: Drawer Spring Mismatch
**What goes wrong:** The drawer animation doesn't use the Soul doc spring (20/100/1.0).
**Why it happens:** @react-navigation/drawer v7 uses hardcoded spring physics (stiffness 1000, damping 500, mass 3) internally. The `drawerType: 'slide'` animation is not configurable via props.
**How to avoid:** Accept the built-in spring for drawer open/close. Focus the "thud" haptic (D-18) on the drawer reaching rest state via gesture events or drawer status callbacks. Do NOT attempt to patch the drawer source.
**Warning signs:** Drawer animation doesn't match Soul doc spec -- this is expected and documented in the `_layout.tsx` comments.

### Pitfall 5: TouchableOpacity vs AnimatedPressable
**What goes wrong:** Some components use opacity feedback instead of scale feedback, breaking the "every press scales to 0.97" contract.
**Why it happens:** Default React Native Pressable doesn't have scale animation. Developers use TouchableOpacity as a quick fix.
**How to avoid:** Audit for any TouchableOpacity usage. Replace with AnimatedPressable + Micro spring scale pattern already established in 8+ components.
**Warning signs:** Tapping a button creates opacity flash instead of satisfying "press" scale-down.

### Pitfall 6: Rim Light Not Visible on Device
**What goes wrong:** The `rgba(255,255,255,0.05)` rim light border is invisible on device.
**Why it happens:** 5% opacity white on a dark surface is extremely subtle. Screen brightness, viewing angle, and fingerprint smudges can make it imperceptible.
**How to avoid:** Start with 0.05 opacity, test on real device at 50% brightness. If invisible, increase to 0.07 or 0.08. Claude's discretion per CONTEXT.md.
**Warning signs:** Side-by-side comparison with ChatGPT shows no depth differentiation on raised surfaces.

   - What's unclear: Exact timing of the `drawerOpen`/`drawerClose` event relative to visual animation completion.
   - Recommendation: Use `addListener('drawerOpen')` and `addListener('drawerClose')` on the drawer navigation to fire Impact Medium haptic. Accept timing may be approximate (within 1-2 frames).
