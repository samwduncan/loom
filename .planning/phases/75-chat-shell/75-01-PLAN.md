    1. `MessageSegment` -- discriminated union type:
    ```typescript
    export type MessageSegment =
      | { type: 'text'; content: string; isStreaming: boolean }
      | { type: 'thinking'; block: ThinkingBlock; isActive: boolean }
      | { type: 'tool_use'; toolCall: ToolCallState }
      | { type: 'code_block'; code: string; language: string; isStreaming: boolean }
      | { type: 'permission'; request: PermissionRequest };
    ```

    2. `parseMessageSegments(content, toolCalls, thinkingBlocks, permissionRequest, isStreaming): MessageSegment[]` -- the core parser that:
       - Splits content on fenced code blocks (``` delimiters) to extract code_block segments with language tag
       - Interleaves tool_use segments at their natural position (after the text that preceded them, in order of `startedAt`)
       - Prepends thinking segments (before main content, per D-05: thinking appears before response text)
       - Appends permission segment at the end if `permissionRequest` is non-null
       - For streaming: the final text segment has `isStreaming: true`; all prior text segments have `isStreaming: false` (ChunkedCodeView pattern per D-18 -- freeze completed blocks, only trailing re-renders)
       - For code_block segments during streaming: if the last code fence is not closed, that code_block segment has `isStreaming: true`

    **Implementation detail for code block extraction:**
    - Use a state machine parser that walks the content string character by character
    - Track whether we're inside a fenced code block (``` boundary)
    - When entering a code fence, capture the language tag (text after ``` on same line, trimmed)
    - When exiting a code fence, emit a code_block segment
    - Text between code blocks becomes text segments
    - Empty text segments (whitespace only between code blocks) should be omitted

    **Import from:**
    - `@loom/shared/types/stream` for `ToolCallState`
    - `@loom/shared/types/message` for `ThinkingBlock`
    - `@loom/shared/stores/stream` for `PermissionRequest` type (import type only)

    **Tool call interleaving strategy (AR fix #10):**
    Tool calls don't have position markers in the content string -- no insertion-point data is available from the backend. Strategy: emit all tool calls as a group after the last text/code segment, before the permission segment. This is a known simplification vs D-04's "at insertion point" spec; actual position data would require backend changes. The visual result is tool chips appearing after the text block, which matches the common case where tool calls follow a text explanation.

    If `toolCalls` array is non-empty and content is also non-empty, tool_use segments appear after all text+code segments. If content is empty (tool-only response), tool_use segments are the only segments.

    **toolCalls parameter typing:** This function accepts both `ToolCallState[]` (from stream store, live streaming) and will also be called with historical `ToolCall[]` from message.toolCalls (see Plan 05).     2. `getRelativeDateSection(date: Date): string` -- per D-09, returns one of 4 section titles:
    ```typescript
    export function getRelativeDateSection(date: Date): string {
      const now = new Date();
      const startOfToday = new Date(now.getFullYear(), now.getMonth(), now.getDate());
      const diffMs = startOfToday.getTime() - new Date(date.getFullYear(), date.getMonth(), date.getDate()).getTime();
      const diffDays = Math.floor(diffMs / 86400000);
      if (diffDays <= 0) return 'Today';  // includes future dates (edge case: clock skew)
      if (diffDays === 1) return 'Yesterday';
      if (diffDays <= 7) return 'Last Week';
      return 'Older';
    }
    ```
    Use start-of-day comparison (not raw timestamp diff) so a session from 11:59 PM yesterday doesn't show as "Today" when viewed at 12:01 AM.

    3. `groupSessionsByDate(sessions: SessionData[]): SectionData[]` -- groups sessions into sections:
    ```typescript
    export function groupSessionsByDate(sessions: SessionData[]): SectionData[] {
      const ORDER = ['Today', 'Yesterday', 'Last Week', 'Older'];
      const groups = new Map<string, SessionData[]>();
      for (const session of sessions) {
        const section = getRelativeDateSection(new Date(session.updatedAt));
        if (!groups.has(section)) groups.set(section, []);
        groups.get(section)!.push(session);
      }
      // Return in fixed order, omitting empty sections
      return ORDER
        .filter(title => groups.has(title))
        .map(title => ({ title, data: groups.get(title)! }));
    }
    ```
    Per D-10: sessions within each section remain sorted by updatedAt descending (the input from useSessions is already sorted).
