Add scroll position save/restore to MessageList.tsx using sessionStorage. The approach:

**Save scroll position (throttled):**
1. Add a `SCROLL_STORAGE_PREFIX` constant: `'loom-scroll-'`.
2. In the existing `handleScroll` callback (around line 114), after computing `distFromBottom`, add a throttled save. Use a module-level `let saveTimeout: ReturnType<typeof setTimeout> | null = null` and inside handleScroll:
   ```
   if (saveTimeout) clearTimeout(saveTimeout);
   saveTimeout = setTimeout(() => {
     sessionStorage.setItem(`${SCROLL_STORAGE_PREFIX}${sessionId}`, String(el.scrollTop));
   }, 200);
   ```
   This throttles writes to 200ms trailing edge — no performance impact.

**Save on session switch (before leaving):**
3. In the session-switch detection block (the `if (prevSession !== sessionId)` block around line 88), BEFORE resetting state, save the current scroll position for the OLD session:
   ```
   const el = scrollRef.current;
   if (el && prevSession) {
     sessionStorage.setItem(`${SCROLL_STORAGE_PREFIX}${prevSession}`, String(el.scrollTop));
   }
   ```

**Restore scroll position after messages load:**
4. Modify the `useLayoutEffect` that scrolls to bottom on session switch (around line 97). Instead of always scrolling to bottom, check sessionStorage first:
   ```
   useLayoutEffect(() => {
     const el = scrollRef.current;
     if (el && messages.length > 0 && scrolledSessionRef.current !== sessionId) {
       scrolledSessionRef.current = sessionId;
       const savedPos = sessionStorage.getItem(`${SCROLL_STORAGE_PREFIX}${sessionId}`);
       requestAnimationFrame(() => {
         if (savedPos !== null) {
           el.scrollTop = Number(savedPos);
         } else {
           el.scrollTop = el.scrollHeight;
         }
       });
     }
   });
   ```
   This means: if we have a saved position, restore it; otherwise scroll to bottom (existing behavior for new sessions).

**Cleanup on session delete:**
5. No cleanup needed here — sessionStorage auto-clears on browser close, and stale entries for deleted sessions are harmless (tiny strings, never read again).

**Why sessionStorage, not localStorage:**
- Scroll positions are ephemeral — valid for current browser tab session only
- Content height changes between browser restarts (new messages from CLI), so old positions would be wrong
- sessionStorage survives page refreshes within the same tab (covers F5 reload case)
- sessionStorage auto-cleans on tab close (no garbage accumulation)
