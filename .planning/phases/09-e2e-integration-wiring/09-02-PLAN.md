    **Create tool-calls.spec.ts (covers COMP-01 + STRM-02 tool channel):**
    Test 1: "tool calls display ToolChip during streaming"
    - Navigate to /chat, wait for WebSocket
    - Send a prompt that WILL trigger tool calls, e.g. "Read the file /etc/hostname and tell me what it says" (Bash/Read tool likely triggered)
    - Wait for `[data-testid="tool-chip"]` to appear (timeout 30s)
    - Assert the ToolChip is visible with a tool name

    Test 2: "tool calls resolve after completion"
    - Continue from test 1 (or new session with same prompt)
    - Wait for streaming to complete (cursor hidden)
    - Assert ToolChip has resolved state (check for resolved visual indicator or absence of pulsing)

    **Create thinking.spec.ts (covers STRM-02 thinking channel):**
    Test 1: "thinking blocks appear during extended reasoning"
    - Navigate to /chat, wait for WebSocket
    - Send a prompt that triggers thinking: "Think carefully step by step: what is 17 * 23?"
    - NOTE: Thinking blocks only appear if the model uses extended thinking. This may not be guaranteed with all prompts. Use `toBeVisible` with a reasonable timeout, and if no thinking block appears, the test should still pass (thinking is model-dependent). Consider using a softer assertion: `expect(thinkingDisclosure).toBeVisible().catch(() => test.skip('No thinking block generated'))` or just check that the response completes without error.
    - If `[data-testid="thinking-disclosure"]` appears, assert it's visible and contains text
    - Assert streaming completes successfully regardless

    **Create scroll-anchor.spec.ts (covers COMP-03 + CLS):**
    Test 1: "auto-scrolls during streaming"
    - Navigate to /chat, wait for WebSocket
    - Send a long prompt: "Write a detailed 500-word essay about the history of computing"
    - Wait for active-message to appear
    - During streaming, check that scroll container's scrollTop is near scrollHeight (auto-scrolling)
    - Use `page.evaluate` to read scrollTop/scrollHeight from the scroll container

    Test 2: "scroll-to-bottom pill appears on manual scroll up"
    - During or after streaming, execute `page.evaluate` to scroll the chat container up by 500px
    - Wait for `[data-testid="scroll-to-bottom-pill"]` to be visible
    - Click the pill
    - Assert pill disappears (auto-scroll re-engaged)

    Test 3: "zero CLS during streaming-to-finalized handoff"
    - Navigate to /chat, wait for WebSocket
    - Inject CLS observer BEFORE streaming:
      ```javascript
      await page.evaluate(() => {
        (window as any).__cls = 0;
        const observer = new PerformanceObserver((list) => {
          for (const entry of list.getEntries()) {
            if (!(entry as any).hadRecentInput) {
              (window as any).__cls += (entry as any).value;
            }
          }
        });
        observer.observe({ type: 'layout-shift', buffered: true });
      });
      ```
    - Send a message, wait for streaming to complete (cursor hidden = finalization happened)
    - Read CLS value: `const cls = await page.evaluate(() => (window as any).__cls);`
    - Assert `cls < 0.1` (Google's "good" threshold)

    **Test file organization notes:**
    - Each spec file should have `test.describe.configure({ mode: 'serial' })` since tests within a file may depend on prior state
    - Use `test.slow()` for tests that need extended timeouts (streaming tests)
    - Clear localStorage in `test.beforeEach` to prevent auth state pollution: `await page.evaluate(() => localStorage.clear())`
    - OR use a shared beforeAll that navigates to /chat once and reuses the session
    - Keep prompts SHORT to minimize API costs and test duration. Exception: scroll anchor test needs a long response.
