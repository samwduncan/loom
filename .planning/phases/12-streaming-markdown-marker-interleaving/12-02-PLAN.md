  <behavior>
    - rehypeToolMarkers transforms a hast text node containing "\x00TOOL:abc123\x00" into [text-before, tool-marker element, text-after]
    - Multiple markers in the same text node are all replaced
    - Markers in different text nodes across the tree are all found
    - Text nodes without markers are left unchanged
    - The tool-marker element has tagName "tool-marker" and properties { "data-id": "abc123" }
    - MarkdownRenderer with rehypeToolMarkers renders tool-marker elements via component override
    - AssistantMessage injects markers into content string at tool positions (between text content and tool call positions)
  </behavior>
  <action>
    1. Create `src/src/lib/rehype-tool-markers.ts`:
       - Import `visit` from `unist-util-visit` and types from `hast`
       - Define `MARKER_REGEX = /\x00TOOL:([^\x00]+)\x00/`
       - Export `rehypeToolMarkers()` function returning a hast transformer
       - Walk tree with `visit(tree, 'text', ...)`. For each text node:
         a. Test if value contains marker regex
         b. If no match, skip (return)
         c. Split text around ALL markers (loop: find marker, split before/after, repeat on after)
         d. Build replacement node array: Text nodes for content, Element nodes with `tagName: 'tool-marker'`, `properties: { 'data-id': toolId }`, `children: []`
         e. Splice replacements into parent.children at the node's index
         f. Return `SKIP` to avoid re-visiting inserted nodes
       - Validate toolId is non-empty string before creating element

    2. Create `src/src/lib/rehype-tool-markers.test.ts`:
       - Test the plugin directly on hast trees (create minimal Root nodes with text children containing markers)
       - Test: single marker splits correctly
       - Test: multiple markers in one text node
       - Test: no markers = no changes
       - Test: marker at start/end of text (empty text nodes not created)
       - Test: nested elements (marker inside a paragraph child)

    3. Update `src/src/components/chat/view/MarkdownRenderer.tsx`:
       - Import `rehypeToolMarkers` from `@/lib/rehype-tool-markers`
       - Import `ToolChip` from `@/components/chat/tools/ToolChip`
       - Add `rehypeToolMarkers` to rehypePlugins: `[rehypeRaw, rehypeToolMarkers]`
       - Add `'tool-marker'` component override to the components object:
         ```typescript
         'tool-marker': ({ node, ...props }: { node?: unknown; 'data-id'?: string }) => {
           const toolId = props['data-id'];
           if (!toolId) return null;
           return <ToolMarkerInline toolCallId={toolId} />;
         },
         ```
       - Accept optional `toolCalls` prop on MarkdownRendererProps (array of ToolCallState for lookup)
       - `ToolMarkerInline` is a small internal component that looks up toolCallId from the toolCalls prop and renders ToolChip

    4. Update `src/src/components/chat/view/AssistantMessage.tsx`:
       - When message has toolCalls, inject `\x00TOOL:id\x00` markers into the content string before passing to MarkdownRenderer
       - Markers are inserted between paragraphs/blocks (after double-newlines) at positions corresponding to each tool call's order
       - Pass `toolCalls` prop (as ToolCallState array) to MarkdownRenderer for the component override lookup
       - Remove the separate tool calls div at the bottom (tool chips now render inline via markers)
       - If no tool call positions can be determined (legacy messages without position data), fall back to appending markers at the end of content

    Note on rehype-raw ordering: `rehypeRaw` processes HTML entities first, then `rehypeToolMarkers` finds null-byte markers in text nodes. Null bytes are not HTML entities so rehype-raw won't interfere.
