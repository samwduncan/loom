<task type="auto" tdd="true">
  <name>Task 2: WebSocket ping/pong heartbeat with silent disconnect detection</name>
  <files>server/index.js, src/src/lib/websocket-client.ts, src/src/lib/websocket-client.test.ts, src/src/lib/websocket-init.ts</files>
  <behavior>
    - Test (backend): server pings all connected WS clients every 15 seconds
    - Test (backend): server terminates clients that miss 2 consecutive pongs (30s)
    - Test (frontend): client resets a pong-timeout timer on each incoming message or pong
    - Test (frontend): client triggers reconnect if no message/pong received for 30 seconds
    - Test (frontend): on WS auth failure (4401 close code or immediate close after connect), client calls refreshAuth() and retries connect with new token
  </behavior>
  <action>
    1. **Server-side ping interval** in `server/index.js`:
       - After `wss.on('connection', ...)`, add a `setInterval` (15s) that iterates `wss.clients`:
         - For each client, check `ws.isAlive` flag. If false, call `ws.terminate()`. If true, set `ws.isAlive = false` and call `ws.ping()`.
       - In the connection handler (`handleChatConnection` and `handleShellConnection`), set `ws.isAlive = true` on connection.
       - Add `ws.on('pong', () => { ws.isAlive = true; })` in both handlers.
       - Clear the interval on server close (`server.on('close', ...)` or `process.on('SIGTERM', ...)`).

    2. **Client-side pong timeout** in `websocket-client.ts`:
       - Add a private `pongTimer: ReturnType<typeof setTimeout> | null = null` field.
       - Add private `resetPongTimer()` method: clears existing timer, sets new 30s timer. On expiry: if state is 'connected', call `this.ws?.close(4000, 'pong timeout')` which triggers handleClose -> scheduleReconnect.
       - Call `resetPongTimer()` in `handleOpen()` and `handleMessage()`.
       - Clear pongTimer in `disconnect()` and when entering 'reconnecting' state.
       - The native browser WebSocket automatically responds to server ping frames with pong, so no explicit pong handling needed on client.

    3. **WS auth failure recovery** in `websocket-init.ts`:
       - In the `onStateChange` handler's 'disconnected' case, check if the close was auth-related (WebSocket close code 4401 or connect failed immediately).
       - If auth-related: call `refreshAuth()` from auth.ts, then `wsClient.tryReconnect()`.
       - Add this as a new private method on WebSocketClient: expose `getLastCloseCode()` returning the last CloseEvent code, set it in `handleClose`.

    4. Update existing `websocket-client.test.ts` with heartbeat timeout tests.
  </action>
