# Phase 67.1 Context: iOS Bug Fixes

## Process Change

Phase 67 shipped gesture code that passes tests but doesn't work on iOS. Root cause: web-first development with iOS validation only at the end.

**67.1 must work differently:**
- Each plan gets device-tested before the next plan starts
- Pine-dove (Mac/Xcode) rebuilds and deploys after each plan
- User tests on real iPhone 16 Pro Max and reports back via relay
- No plan is "autonomous: true" — every plan has a device checkpoint
- If something doesn't work on iOS, we fix it before moving on

## Key Technical Lessons from Phase 67

1. **Radix ContextMenu does NOT work on iOS WKWebView.** iOS doesn't fire `contextmenu` on long-press. Need custom long-press detection or a different UI pattern entirely (e.g., visible action buttons, bottom sheet on long-press via touch timer).

2. **@use-gesture/react swipe conflicts with sidebar touch handlers.** The sidebar's raw touchStart/touchMove/touchEnd handlers are too sensitive and capture horizontal swipes before useDrag gets them. Need gesture zone isolation (e.g., edge-only for sidebar, interior for swipe-to-delete).

3. **Haptics module may not be initialized.** Check if native-plugins.ts actually calls setHapticsModule(). Verify in Xcode console.

4. **CSS/jsdom tests prove nothing about iOS behavior.** Gesture interactions, haptics, native plugins, and WKWebView rendering all require real device testing.

5. **Page reloads (#5) may be WKWebView memory eviction or Capacitor live reload.** Needs Xcode console investigation — look for `didReceiveMemoryWarning`, `webViewWebContentProcessDidTerminate`, or Capacitor reload logs.
