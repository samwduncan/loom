# Phase 67.1: iOS Bug Fixes - Research

**Researched:** 2026-03-29
**Domain:** iOS WKWebView / Capacitor / Touch Gestures / CSS Tokens
**Confidence:** HIGH

## Summary

Phase 67.1 fixes 15 Forgejo issues (#2-#17) discovered during real-device testing of Phase 67's iOS gesture code. The bugs fall into five categories: (1) Radix ContextMenu fundamentally broken on iOS WKWebView because iOS does NOT fire `contextmenu` events on long-press, (2) gesture zone conflicts between sidebar swipe-to-close and session swipe-to-delete, (3) Capacitor plugin configuration errors (haptics not firing, accessory bar visible, keyboard desync), (4) undefined CSS custom properties causing silent fallback failures, and (5) UX refinements (auto-focus keyboard, PTR animation, horizontal overflow, UI density).

The critical finding is that **Radix ContextMenu cannot be fixed on iOS** -- it relies on the browser `contextmenu` event which iOS Safari/WKWebView has never fired on long-press (confirmed since iOS 13, still true in iOS 17+). The solution requires replacing Radix ContextMenu with a custom long-press detection mechanism on mobile that programmatically opens a popover/bottom sheet.

**Primary recommendation:** Replace Radix ContextMenu usage on mobile with a custom `useLongPress` hook + popover pattern. Fix the 5 CSS undefined token errors first (zero-risk). Then address gesture conflicts and Capacitor plugin issues in priority order. Every plan must be device-tested before the next plan starts.

## Architecture Patterns

### Pattern 1: Custom Long-Press for iOS (Replaces Radix ContextMenu on Mobile)

**What:** iOS WKWebView does NOT fire the `contextmenu` event on long-press. This is a fundamental WebKit limitation since iOS 13 (2019), confirmed still present in iOS 17+. Radix ContextMenu depends entirely on this event. The fix requires a custom long-press detection mechanism on mobile that falls back to Radix on desktop.

**Approach:** Create a `useLongPress` hook that uses `touchstart`/`touchend`/`touchmove` timers to detect a 500ms press, then programmatically opens a Radix Popover (not ContextMenu). On desktop, keep Radix ContextMenu as-is (right-click works).

**Implementation pattern:**
```typescript
// src/hooks/useLongPress.ts
import { useRef, useCallback } from 'react';
import { IS_NATIVE } from '@/lib/platform';
import { useMobile } from '@/hooks/useMobile';

interface UseLongPressOptions {
  delay?: number;       // default 500ms
  onLongPress: () => void;
  onCancel?: () => void;
  moveThreshold?: number; // cancel if finger moves > N px (default 10)
}

export function useLongPress({ delay = 500, onLongPress, onCancel, moveThreshold = 10 }: UseLongPressOptions) {
  const timerRef = useRef<ReturnType<typeof setTimeout>>();
  const startPos = useRef({ x: 0, y: 0 });
  const isMobile = useMobile();

  const handleTouchStart = useCallback((e: React.TouchEvent) => {
    const touch = e.touches[0];
    if (!touch) return;
    startPos.current = { x: touch.clientX, y: touch.clientY };
    timerRef.current = setTimeout(() => {
      onLongPress();
    }, delay);
  }, [delay, onLongPress]);

  const handleTouchMove = useCallback((e: React.TouchEvent) => {
    const touch = e.touches[0];
    if (!touch || !timerRef.current) return;
    const dx = Math.abs(touch.clientX - startPos.current.x);
    const dy = Math.abs(touch.clientY - startPos.current.y);
    if (dx > moveThreshold || dy > moveThreshold) {
      clearTimeout(timerRef.current);
      timerRef.current = undefined;
      onCancel?.();
    }
  }, [moveThreshold, onCancel]);

  const handleTouchEnd = useCallback(() => {
    if (timerRef.current) {
      clearTimeout(timerRef.current);
      timerRef.current = undefined;
    }
  }, []);

  // Only bind touch events on mobile -- desktop uses Radix ContextMenu
  if (!isMobile) return {};

  return {
    onTouchStart: handleTouchStart,
    onTouchMove: handleTouchMove,
    onTouchEnd: handleTouchEnd,
  };
}
```

**CSS requirement for long-press triggers:**
```css
/* Prevent iOS text selection and callout on long-press targets */
[data-long-press-target] {
  -webkit-touch-callout: none;
  -webkit-user-select: none;
  user-select: none;
}
```

**CRITICAL:** For SessionItemContextMenu, text selection is acceptable on the content (titles). But for MessageContextMenu, the user explicitly wants text selection preserved on message content. The long-press must suppress selection ONLY during the press timer, then re-enable. Alternative: use a dedicated long-press "grip zone" (e.g., a small icon area) rather than the entire message surface.

**Confidence:** HIGH -- This is a well-documented iOS limitation. Custom long-press timers are the standard workaround in every Capacitor/Ionic app.

### Pattern 2: Gesture Zone Isolation (Sidebar Edge vs Interior Swipe)

**What:** The sidebar's raw `touchStart`/`touchMove`/`touchEnd` handlers in `Sidebar.tsx` capture ALL horizontal swipes on the sidebar panel, preventing `useSwipeToDelete` from working on session items inside the panel.

**Root cause:** The sidebar panel's `onTouchMove` handler (line 67-75 of Sidebar.tsx) activates swipe tracking when `deltaX < -10` -- this is a 10px threshold with NO horizontal zone restriction. Any leftward swipe anywhere on the sidebar panel triggers the sidebar close animation.

**Fix strategy:** The sidebar close gesture should ONLY activate from the left edge of the sidebar panel (first 20-30px). Interior swipes should fall through to child gesture handlers (useSwipeToDelete).

```typescript
// In Sidebar.tsx handleTouchStart:
const EDGE_ZONE_WIDTH = 30; // pixels from right edge of sidebar panel
const sidebarWidth = sidebarPanelRef.current?.offsetWidth ?? 280;
const touchXRelative = touch.clientX;

// Only activate sidebar-close swipe from the right edge of the sidebar
// (the part closest to the main content area)
touchRef.current.isEdgeSwipe = touchXRelative > (sidebarWidth - EDGE_ZONE_WIDTH);

// In handleTouchMove, only track if isEdgeSwipe:
if (!touchRef.current.isEdgeSwipe) return; // Let child handlers process
```

**Confidence:** HIGH -- Standard gesture zone isolation pattern. The fix is straightforward.

### Pattern 3: Delete Button Tap Interception Fix (#17)

**What:** The delete button is visible but not tappable because `useDrag` from `@use-gesture/react` captures touch events on the entire session item div, including the revealed delete button area.

**Root cause analysis:** In `SessionItem.tsx`, the `{...bind()}` spread is on the content div (line 178), which translates via transform. The delete button is positioned absolute behind it (line 159-175). When the content slides left, the delete button IS exposed BUT the content div's gesture handler still covers the tap area because the content div hasn't been clipped.

**Fix:** The delete button container needs to be OUTSIDE the gesture-bound element, or the gesture-bound element needs `pointer-events: none` on the revealed portion. The cleanest fix: move the delete button to be a sibling that's absolutely positioned and rendered above the gesture layer when revealed.

```typescript
// Restructure: Delete button at z-index above gesture layer
<div className="relative overflow-hidden">
  {/* Gesture-bound content (slides left) */}
  <div {...(isMobile ? bind() : {})} style={{ transform: ... }}>
    {/* Session content */}
  </div>
  {/* Delete button -- OUTSIDE gesture binding, absolutely positioned */}
  {isMobile && revealed && (
    <button
      onClick={handleSwipeDelete}
      className="absolute right-0 top-0 bottom-0 w-[80px] z-10 ..."
      style={{ pointerEvents: 'auto' }}
    >
      Delete
    </button>
  )}
</div>
```

**Confidence:** HIGH -- Standard z-ordering fix for swipe-to-delete patterns.

### Pattern 5: Keyboard Animation Sync (#3)

**What:** Visible delay between keyboard appearing and composer sliding up. The current `useKeyboardOffset` hook (line 30-31) is a NO-OP on native because it relies on WKWebView's default Body resize. But the default Body resize animates AFTER the keyboard animation finishes, not in sync.

**Fix approach:** Switch from native Body resize to manual keyboard offset management using Capacitor Keyboard events:

```typescript
// useKeyboardOffset.ts -- native path (currently no-op)
if (IS_NATIVE) {
  const mod = await nativePluginsReady.then(() => getKeyboardModule());
  if (!mod) return;
  const { Keyboard } = mod;

  // Use keyboardWillShow (not keyboardDidShow) to start animation before keyboard finishes
  const willShowHandle = await Keyboard.addListener('keyboardWillShow', (info) => {
    document.documentElement.style.setProperty(
      '--keyboard-offset',
      `${info.keyboardHeight}px`
    );
  });

  const willHideHandle = await Keyboard.addListener('keyboardWillHide', () => {
    document.documentElement.style.setProperty('--keyboard-offset', '0px');
  });

  return () => {
    willShowHandle.remove();
    willHideHandle.remove();
  };
}
```

**Also requires:** Setting `KeyboardResize` to `None` in native-plugins.ts so WKWebView doesn't double-resize:
```typescript
await Keyboard.setResizeMode({ mode: KeyboardResize.None });
```

**Confidence:** MEDIUM -- This is the documented approach for Capacitor keyboard sync, but it reverses a prior decision (Phase 59/60 chose native resize to avoid race conditions). Must be device-tested carefully.

### Anti-Patterns to Avoid

- **Do NOT add a `contextmenu` event polyfill** -- iOS will never fire this event. Custom timer is the only path.
- **Do NOT apply `user-select: none` globally on messages** -- text selection must work on message content. Only suppress during active long-press timer.
- **Do NOT use `KeyboardResize.Body`** -- this resizes the HTML body element which causes layout shifts in the Zustand-driven React tree.
- **Do NOT move ALL gesture detection to @use-gesture** -- the sidebar close swipe is intentionally raw touch handlers because @use-gesture's useDrag doesn't support the "animate-out-then-close" pattern cleanly.

## Common Pitfalls

### Pitfall 1: iOS contextmenu Event Assumption
**What goes wrong:** Radix ContextMenu works perfectly in Chrome DevTools mobile emulator but fails on real iOS device.
**Why it happens:** Chrome emulator fires `contextmenu` on simulated long-press. iOS Safari/WKWebView never fires this event.
**How to avoid:** Always use custom long-press timer on mobile. Test on real device, not emulator.
**Warning signs:** "Works in DevTools" but "nothing happens on phone."

### Pitfall 2: Text Selection vs Long-Press Conflict
**What goes wrong:** Adding `-webkit-user-select: none` to fix long-press breaks the ability to select/copy text from messages.
**Why it happens:** iOS triggers text selection on the same gesture (touch-and-hold) that we want to use for context menus.
**How to avoid:** Only suppress selection on the long-press trigger element, not on message content. Use `e.preventDefault()` in the touchstart handler during the timer, then re-enable on cancel. Or: designate a specific "grip zone" for long-press that doesn't overlap the text area.
**Warning signs:** Users can't copy text from messages anymore.

### Pitfall 3: WKWebView Process Termination Reloads (#5)
**What goes wrong:** Page randomly reloads during active chat, losing streaming context.
**Why it happens:** iOS kills the WKWebView web content process under memory pressure (`webViewWebContentProcessDidTerminate`). Capacitor 3.4.1+ auto-reloads on this event. Also possible: Capacitor live reload config in server.url mode with HMR.
**How to avoid:**
  1. Check Capacitor config for live reload settings
  2. Monitor memory usage in Xcode Instruments
  3. Minimize JS memory footprint (WeakRefs for large objects)
  4. The Capacitor reload-on-terminate is actually the correct behavior -- the issue is that app state is lost
  5. Ensure WebSocket auto-reconnect (GESTURE-08 already handles this)
**Warning signs:** White screen flash followed by app restart. Check Xcode console for "webViewWebContentProcessDidTerminate".
**Note:** This may not be fixable in web code alone. If memory pressure is the cause, the fix is reducing memory usage. If it's Capacitor live reload, the fix is configuration.

### Pitfall 4: Gesture Handler Event Swallowing
**What goes wrong:** @use-gesture useDrag's `filterTaps: true` should let taps through, but the gesture handler's pointer event capture prevents child elements from receiving clicks.
**Why it happens:** The gesture-bound element's touch handlers call `e.preventDefault()` or `e.stopPropagation()` internally, or the element overlaps the child target.
**How to avoid:** Ensure the delete button is NOT a child of the gesture-bound element. Use sibling positioning with z-index.
**Warning signs:** Console shows click events on parent but not on child button.

### Pitfall 5: Double Keyboard Resize
**What goes wrong:** Switching from native resize to manual keyboard offset causes the viewport to resize AND the content to shift, creating a double-animation effect.
**Why it happens:** WKWebView's default Body resize mode adjusts the viewport simultaneously with the CSS --keyboard-offset transition.
**How to avoid:** When using manual keyboard offset, MUST set `KeyboardResize.None` in Capacitor config. These approaches are mutually exclusive.
**Warning signs:** Content jumps down then bounces back up when keyboard opens.

## Open Questions

1. **WKWebView page reloads (#5) -- is this memory pressure or live reload?**
   - What we know: Capacitor auto-reloads on `webViewWebContentProcessDidTerminate`. Server.url mode may also trigger HMR reloads.
   - What's unclear: Which mechanism is causing the reloads. Need Xcode console logs.
   - Recommendation: Add diagnostic logging. Check `capacitor.config.ts` server settings. If memory pressure, may need to defer to later milestone. If live reload, config fix.

2. **Keyboard animation sync approach -- revert to KeyboardResize.None?**
   - What we know: Phase 59/60 intentionally chose native resize to avoid race conditions. Switching to manual offset requires reversing this decision.
   - What's unclear: Whether the race condition from Phase 59/60 will resurface.
   - Recommendation: Try `keyboardWillShow` events first WITHOUT changing resize mode. If the default resize plus manual offset causes double-animation, then switch to KeyboardResize.None.

3. **Haptics -- is the issue devDependencies or something else?**
   - What we know: @capacitor/haptics is in devDependencies, not dependencies. The Phase 67 decision to move all plugins to production deps was apparently not completed.
   - What's unclear: Whether Vite tree-shakes devDeps from production builds (it shouldn't, since dynamic import resolves from node_modules regardless).
   - Recommendation: Move to production deps as a first step. Add console.log diagnostics if still not working.

4. **Message long-press -- how to preserve text selection?**
   - What we know: Messages must remain selectable (CRITICAL decision from Phase 67). But long-press is the same gesture as text selection on iOS.
   - What's unclear: Whether users can tolerate a slightly different interaction (e.g., visible action button instead of long-press).
   - Recommendation: Use `-webkit-user-select: none` + `-webkit-touch-callout: none` on the trigger wrapper, but provide a separate "select text" action in the context menu/popover that re-enables selection. Or: use a visible action icon (three dots) for message actions on mobile instead of long-press.
