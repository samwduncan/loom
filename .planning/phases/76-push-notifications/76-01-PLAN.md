*CRITICAL: Push triggers that fire REGARDLESS of WS connection state (fixes BLOCKER 1):*

Do NOT wire push triggers to the WS proxy's `send()` method. When a WS client disconnects (the PRIMARY push use case per D-01), the `WebSocketWriter.send()` silently drops messages because `ws.readyState !== OPEN`.

Instead, modify `WebSocketWriter` to ALSO emit push trigger events even when the WS is closed:

```javascript
class WebSocketWriter {
    constructor(ws, userId) {
        this.ws = ws;
        this.userId = userId;  // Store userId for push triggers
        this.sessionId = null;
        this.isWebSocketWriter = true;
    }

    send(data) {
        // Send via WS if connected
        if (this.ws.readyState === 1) {
            this.ws.send(JSON.stringify(data));
        }

        // ALWAYS trigger push service, regardless of WS state.
        // PushService will check if user is actively viewing -- if WS is open
        // and user is foregrounded, push is suppressed. If WS is closed,
        // push fires (which is the primary use case per D-01).
        this._triggerPushIfNeeded(data);
    }

    _triggerPushIfNeeded(data) {
        if (!this.userId) return;

        if (data.type === 'claude-complete') {
            // [SS-1] Resolve human-readable session name from cache DB
            const sessionName = pushService.getSessionName(data.sessionId);
            pushService.notifySessionComplete(
                this.userId, data.sessionId, sessionName,
                data.exitCode === 0, null
            );
        } else if (data.type === 'claude-error') {
            // [SS-1] Resolve human-readable session name from cache DB
            const sessionName = pushService.getSessionName(data.sessionId);
            pushService.notifySessionComplete(
                this.userId, data.sessionId, sessionName,
                false, data.error
            );
        } else if (data.type === 'claude-permission-request') {
            // [SS-1] Resolve human-readable session name from cache DB
            const sessionName = pushService.getSessionName(data.sessionId);
            pushService.notifyPermissionRequest(
                this.userId, data.sessionId, sessionName,
                data.requestId, data.toolName,
                JSON.stringify(data.input).slice(0, 80)
            );
        }
    }

    // ... existing setSessionId, getSessionId methods unchanged
}
```

Update the `WebSocketWriter` instantiation in `handleChatConnection` to pass userId:


This ensures push triggers fire for EVERY session event -- the PushService's `shouldSuppressPush()` method handles the logic of whether to actually send the notification (it checks if the user has any active, foregrounded clients viewing the session).

**Why this works:** When the mobile app is backgrounded and WS disconnects, the SDK query continues running server-side.
