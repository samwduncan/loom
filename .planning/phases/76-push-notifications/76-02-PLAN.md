`consumeColdStartResponse(): Notifications.NotificationResponse | null`
- If `hasBeenConsumed` return null. Otherwise set `hasBeenConsumed = true` and return `coldStartResponse`. Per D-13: prevents double navigation.

`async getInitialNotificationResponse(): Promise<Notifications.NotificationResponse | null>`
- First try `consumeColdStartResponse()`. If non-null, return it.
- Fallback: `await Notifications.getLastNotificationResponseAsync()`. Return result.
- Per Pitfall 1 and 2 in RESEARCH.md.

`async registerNotificationCategories(): Promise<void>`
- Per D-05, register two categories:
  - `'session_complete'` with action: `{ identifier: 'open_session', buttonTitle: 'Open Session', options: { opensAppToForeground: true } }`
  - `'permission_request'` with actions: `{ identifier: 'approve', buttonTitle: 'Approve', options: { opensAppToForeground: false, isDestructive: false } }` and `{ identifier: 'deny', buttonTitle: 'Deny', options: { opensAppToForeground: false, isDestructive: true } }`
- Per Pitfall 5: must be called before any push can arrive.

`async registerForPushNotifications(): Promise<string | null>`
- Check `Device.isDevice` -- if false, `console.warn('[Push] Not a physical device')` and return null.
- `getPermissionsAsync()` -> if not 'granted', `requestPermissionsAsync({ ios: { allowAlert: true, allowBadge: true, allowSound: true } })`.
- If still not granted, return null.
- Get `projectId` from `Constants.expoConfig?.extra?.eas?.projectId`. If missing, `console.error('[Push] No EAS projectId')` and return null.
- `const tokenData = await Notifications.getExpoPushTokenAsync({ projectId })`.
- Return `tokenData.data` (string like `"ExponentPushToken[...]"`).

**useEffect 1: Registration (runs once on mount):**
1. Call `registerNotificationCategories()` (per Pitfall 5: before any push can arrive).
2. Call `registerForPushNotifications()` -> get token.
3. If token: store in `pushTokenRef.current`, register with backend: `apiClient.post('/api/push/register', { token, notificationMode: getNotificationMode() })`.
4. Set up foreground notification handler via `Notifications.setNotificationHandler()`:
   ```typescript
   Notifications.setNotificationHandler({
     handleNotification: async (notification) => {
       const data = notification.request.content.data as Record<string, unknown>;
       // Per D-01 + Pitfall 7: client-side gate as belt-and-suspenders
       // If this is the session we're viewing, suppress banner
       // For now, always show (the server-side check is primary)
       return {
         shouldShowBanner: true,
         shouldShowList: true,
         shouldPlaySound: true,
         shouldSetBadge: true,
       };
     },
   });
   ```

**useEffect 2: Notification response listener (warm launch):**
- `Notifications.addNotificationResponseReceivedListener((response) => { handleNotificationResponse(response); });`
- Return cleanup: `subscription.remove()`.

**useEffect 3: Cold-start deep link (runs once after mount):**
- If `hasHandledColdStartRef.current` return.
- Call `getInitialNotificationResponse()`. If non-null: `handleNotificationResponse(response)`.

**useEffect 4: AppState listener for badge clear and queue drain:**
- Listen to `AppState.addEventListener('change', handler)`.
- On `'active'` state: 
  - Clear badge: `Notifications.setBadgeCountAsync(0)`.
  - Drain pending approvals queue (but only if WS is connected -- see below).

**useEffect 5: WS connection state -> queue drain:**
- Subscribe to `useConnectionStore` state. When `status` transitions to `'connected'`:
  - Read `getPendingApprovals()`. For each approval, send via WS: `{ type: 'claude-permission-response', requestId: approval.requestId, allow: approval.action === 'approve' }`.
  - After all sent, `clearPendingApprovals()`.
  - Per D-07: if any response fails (backend returns "expired"), show toast "Permission request expired".
  - Per Pitfall 3: never drain until WS is connected.

**handleNotificationResponse function:**
```typescript
function handleNotificationResponse(response: Notifications.NotificationResponse) {
  const data = response.notification.request.content.data as Record<string, unknown>;
  const actionId = response.actionIdentifier;

  // Permission action buttons (D-05, D-06)
  if (actionId === 'approve' || actionId === 'deny') {
    const requestId = data.requestId as string;
    if (requestId) {
      queueApproval(requestId, actionId as 'approve' | 'deny');
    }
    // Don't navigate for background actions
    if (actionId === 'deny') return; // deny = no navigation
    if (actionId === 'approve') return; // approve = no navigation (opensAppToForeground: false)
  }

  // Deep link navigation (D-09, D-10, D-11, D-12)
  const sessionId = data.sessionId as string;
  if (sessionId) {
    hasHandledColdStartRef.current = true;
    // D-12: validate session exists before navigating
    apiClient.get(`/api/projects`).then(() => {
      // Simple existence check -- navigate to session
      router.push(`/(drawer)/(stack)/chat/${sessionId}`);
    }).catch(() => {
      showToast('Session no longer available', { duration: 3000 });
    });
  }
}
```

NOTE: The deep link validation (D-12) should ideally check if the specific session exists. The simplest approach is to navigate and let the chat screen handle a missing session gracefully. A more robust approach: call a session-specific endpoint. For now, navigate directly and handle 404 in the chat screen -- add a comment noting this.
