Build the mobile notification infrastructure: cold-start safe listeners, MMKV preference/queue storage (including push token persistence), the useNotifications hook, and the in-app NotificationBanner component.

Purpose: These are the mobile-side building blocks that all push notification features depend on. The notifications lib handles the tricky cold-start listener registration at module scope -- critically, the module-level listener ONLY stores the response and never calls API or navigates (auth may not be ready). The push-preferences lib provides durable MMKV storage for notification mode preferences, the offline permission-approval queue, AND push token persistence. The useNotifications hook orchestrates token registration, listener setup, deep linking, and queue drain. The NotificationBanner provides in-app notification display when foreground.

**handleNotificationResponse function (D-12 session validation fixed per checker WARNING 5):**
```typescript
function handleNotificationResponse(response: Notifications.NotificationResponse) {
  const data = response.notification.request.content.data as Record<string, unknown>;
  const actionId = response.actionIdentifier;

  // Permission action buttons (D-05, D-06)
  if (actionId === 'approve' || actionId === 'deny') {
    const requestId = data.requestId as string;
    if (requestId) {
      queueApproval(requestId, actionId as 'approve' | 'deny');
    }
    // Don't navigate for background actions
    if (actionId === 'deny') return; // deny = no navigation
    if (actionId === 'approve') return; // approve = no navigation (opensAppToForeground: false)
  }

  // Deep link navigation (D-09, D-10, D-11, D-12)
  const sessionId = data.sessionId as string;
  if (sessionId) {
    hasHandledColdStartRef.current = true;

    // [S-8] This function is ONLY called from useEffect 2 (warm) or useEffect 3 (cold).
    // Both run inside AuthenticatedApp, so apiClient has a valid JWT.
    // D-12: Validate session exists by fetching projects and searching for the sessionId.
    apiClient.get('/api/projects').then((projects: any[]) => {
      // Search all projects' sessions for the target sessionId
      const sessionExists = projects.some((project: any) =>
        project.sessions?.some((s: any) => s.id === sessionId)
      );

      if (sessionExists) {
        router.push(`/(drawer)/(stack)/chat/${sessionId}`);
      } else {
        // Session not found in any project -- navigate to session list with toast
        router.replace('/(drawer)');
        showToast('Session no longer available', { duration: 3000 });
      }
    }).catch(() => {
      // Server unreachable -- navigate optimistically, chat screen handles 404
      // Better to show stale data than block navigation entirely
      router.push(`/(drawer)/(stack)/chat/${sessionId}`);
    });
  }
}
```

**Key difference from original (fixes WARNING 5):** The original used `apiClient.get('/api/projects')` as a generic reachability check, which only proved the server was up, not that the session existed. The fix fetches the project list and searches for the specific sessionId. If the session is not found, navigates to the drawer (session list) and shows a toast per D-12. If the server is unreachable (catch), navigates optimistically (the chat screen can handle missing sessions gracefully).
