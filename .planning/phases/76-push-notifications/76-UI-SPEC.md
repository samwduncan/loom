## Screen Specifications

### Notification Settings Screen

**Layout:**
- Full-screen pushed onto navigation stack from drawer (Navigation spring, horizontal slide from right).
- Navigation header: "Notifications" (17px SemiBold, centered), back arrow (left).
- Background: `surface-base`.
- Content: ScrollView with grouped sections.

**Sections:**

1. **"Notification Mode" section:**
   - Section header: "NOTIFICATION MODE" (12px Caption, `text-muted`, uppercase, px-16).
   - 4 radio-style rows, 56px min-height each, `surface-raised` background, px-16 py-16:
     - "All notifications" -- body text left, checkmark right when selected.
     - "Failures and permissions" -- body text left, checkmark right when selected.
     - "Permissions only" -- body text left, checkmark right when selected.
     - "None" -- body text left, checkmark right when selected.
   - Section footer: "Choose which events trigger push notifications." (13px Small, `text-muted`, px-16, pt-8).
   - Selected row shows accent-colored checkmark (SF Symbol `checkmark`, 17px, accent color).
   - Unselected rows show no checkmark.

2. **"Behavior" section:**
   - Section header: "BEHAVIOR" (12px Caption, `text-muted`, uppercase, px-16).
   - "Batch notifications" toggle row: title "Batch completions", subtitle "Group session completions within 5 seconds" (13px Small, `text-muted`), toggle right.
   - Section footer: none.

**Motion:**
- Screen entrance: Navigation spring (22/130, ~300ms, horizontal slide from right).
- Row press: Micro spring scale (0.97) + haptic Impact Light.
- Toggle switch: Micro spring on thumb position. Haptic Selection on toggle.
- Checkmark appear/disappear: FadeIn/FadeOut 150ms.

**Color:**
- Standard surface hierarchy. No dynamic color shifts.
- Selected row: no background change -- checkmark is the only indicator.
- Toggle active: accent color thumb/track.
- Toggle inactive: `surface-overlay` thumb, `border-subtle` track.

**Source:** D-14, D-15, D-16 from CONTEXT.md. NATIVE-APP-SOUL.md Settings section.

### In-App Notification Banner

**Layout:**
- Floating banner below status bar, overlapping content.
- Background: glass surface (expo-blur `intensity={40}`, tint `dark`, overlay `rgba(0,0,0,0.35)`).
- Border: 1px `border-subtle`, rounded-2xl (16px radius).
- Horizontal margin: 16px (md) from screen edges.
- Content row: provider icon (24px, left) + text column (title + body) + dismiss area (right).
- Title: session name (15px Body, SemiBold, `text-primary`, single line, truncated).
- Body: event summary (13px Small, `text-secondary`, max 2 lines, truncated).
- Min-height: 56px. Vertical padding: 12px.

**Permission banner variant:**
- Same layout plus warm accent tint: `accent` at 0.1 opacity layered behind glass.
- Body text includes tool name and truncated description.
- No action buttons in banner -- tap to navigate to session where inline PermissionCard handles it.

**Error banner variant:**
- Same layout plus destructive tint: `destructive` at 0.1 opacity layered behind glass.

**Session-complete banner variant:**
- Standard glass treatment, no color tint.
- Body: "Session completed successfully" or "Session failed: {error summary}".

**Batched banner variant:**
- Title: "Sessions completed" (15px Body, SemiBold).
- Body: "{N} sessions finished" (13px Small, `text-secondary`).
- Tapping navigates to session list (drawer opens).

**Motion:**
- Entrance: Navigation spring (22/130, ~300ms) slide-down from above screen (translateY -80 to 0).
- Auto-dismiss: 4 seconds, then slide-up with Standard spring (20/150, ~250ms, translateY 0 to -80, opacity 1 to 0).
- Permission banners: 8 seconds auto-dismiss (longer because actionable).
- Tap: Micro spring scale (0.97) + haptic Impact Light, then navigates and dismisses.
- Swipe up to dismiss: gesture-tracked translateY, spring to dismissed position on release if velocity > 200 pts/sec.

**Color:**
- Per NATIVE-APP-SOUL.md Notification UI section (pre-populated above).

**Source:** NATIVE-APP-SOUL.md Notification UI section. D-23 from CONTEXT.md.

### Push Notification (Native iOS)

**Not custom UI** -- standard iOS notification banner. Contract covers content formatting only.

**Notification categories:**

| Category ID | Actions | Req |
|-------------|---------|-----|
| `session_complete` | "Open Session" (foreground) | PUSH-01, PUSH-02 |
| `permission_request` | "Approve" (background, destructive: false), "Deny" (background, destructive: true) | PUSH-03, PUSH-04 |

**Payload format** (from D-21):
```
{
  title: "{session name}",
  body: "{event-specific summary}",
  data: {
    type: "session_complete" | "permission_request",
    sessionId: "{uuid}",
    requestId: "{uuid}",  // permission_request only
    url: "loom://chat/{sessionId}"
  },
  categoryId: "{category}",
  sound: "default",
  badge: {increment}
}
```

**Notification body copy:**

| Event | Title | Body |
|-------|-------|------|
| Session complete (success) | "{session name}" | "Session completed" |
| Session complete (error) | "{session name}" | "Session failed: {first 80 chars of error}" |
| Permission request | "{session name}" | "{toolName}: {description, max 80 chars}" |
| Batched completions | "Sessions completed" | "{N} sessions finished" |

**Sound:** Default system sound. Custom sound deferred (Claude's Discretion from CONTEXT.md).

**Badge:** Increment on each notification. Clear to 0 on app foreground (AppState `active` listener).

**Source:** D-05, D-06, D-21 from CONTEXT.md.

## Interaction States

### Notification Permission Flow

| State | Visual | Action |
|-------|--------|--------|
| Not prompted | No indicator | App calls `Notifications.requestPermissionsAsync()` after auth on first launch |
| Granted | Push token registered, settings screen fully interactive | Normal operation |
| Denied | Settings screen shows "Notifications are disabled in system settings" banner at top, accent "Open Settings" link | Tapping "Open Settings" opens iOS Settings via `Linking.openSettings()` |
| Provisional | Same as Granted (iOS delivers quietly) | Normal operation |

### Deep Link States

| State | Behavior | Source |
|-------|----------|--------|
| Warm launch (app backgrounded) | Expo Router handles `data.url` automatically | D-10 |
| Cold launch (app killed) | `Notifications.getLastNotificationResponseAsync()` in root layout after auth, manual `router.push()` | D-11 |
| Session exists | Navigate to `(drawer)/(stack)/chat/[id]` | D-09 |
| Session deleted/missing | Navigate to session list + toast "Session no longer available" | D-12 |
| Double-navigation guard | `hasHandledColdStartNotification` ref prevents duplicate | D-13 |

### Offline Approval Queue (MMKV)

| State | Behavior | Source |
|-------|----------|--------|
| User taps Approve/Deny on lock screen | Store `{ requestId, action, timestamp }` in MMKV queue | D-06 |
| App returns to foreground | Drain MMKV queue, send responses via WebSocket | D-24 |
| Agent timeout before delivery | Backend returns "expired", app shows toast "Permission request expired" | D-07 |
| Duplicate response | Backend validates requestId exists and is pending, rejects duplicates | D-08 |
