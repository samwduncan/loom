  <behavior>
    - Test 1: Two concurrent GET requests to the same URL return the same promise (only one fetch call)
    - Test 2: A GET request after the first one resolves fires a new fetch (not cached permanently)
    - Test 3: POST/PATCH/DELETE requests are never deduplicated (mutation safety)
    - Test 4: If the shared promise rejects, all waiters receive the error and the entry is cleaned up
    - Test 5: Requests with different URLs are not deduplicated
  </behavior>
  <action>
    Add a module-level `Map<string, Promise<Response>>` called `inflightRequests` to api-client.ts.

    Dedup logic (only for GET requests, which is the default when no method is specified):
    1. Compute a cache key from `path` (GET requests don't have meaningful body differences in this app)
    2. If key exists in map, return the existing promise (clone the response so each caller gets its own body)
    3. If not, create the fetch, store the promise in the map, and delete the entry in a `.finally()` block
    4. Only dedup when `options.method` is undefined or 'GET' -- never dedup POST/PATCH/DELETE

    Implementation detail: The dedup map stores the raw `fetch()` Promise<Response>, not the parsed JSON. Each consumer clones via `response.clone()` so both can read the body. The `.finally()` cleanup ensures entries don't leak.

    The existing 401-retry logic stays intact -- dedup wraps around the outer apiFetch, not the inner doFetch.

    Create api-client.test.ts with vitest, mocking global.fetch via vi.fn(). Test all 5 behaviors above.
  </action>
