**useUsageMetrics.ts changes (per D-12):**

1. Replace import: change `import { getToken } from '@/lib/auth';` to `import { apiFetch } from '@/lib/api-client';`
2. Rewrite `fetchMetricsData()` (lines 37-48):
   ```typescript
   async function fetchMetricsData(): Promise<UsageMetrics | null> {
     try {
       return await apiFetch<UsageMetrics>('/api/usage/metrics');
     } catch {
       // Silently fail -- metrics are non-critical
       return null;
     }
   }
   ```
   This eliminates the manual `getToken()` + header injection (Pitfall 6 from research -- apiFetch handles auth headers).

3. Rewrite `useSessionTokenUsage` fetch (lines 92-102):
   Replace the manual `fetch()` with `getToken()` header construction:
   ```typescript
   apiFetch<{ used: number; total: number }>(
     `/api/projects/${projectName}/sessions/${sessionId}/token-usage`,
   )
     .then((data) => {
       if (!cancelled && typeof data.used === 'number') {
         setUsage({ used: data.used, total: data.total });
       }
     })
     .catch(() => {});
   ```
   Note: apiFetch already handles auth headers and JSON parsing. The `.json()` call is no longer needed since apiFetch returns parsed JSON.

4. Remove the `getToken` import entirely -- it's no longer used in this file.

**Critical detail:** `apiFetch` throws on non-ok responses (processResponse checks res.ok). The current code checks `res.ok` manually. After migration, a non-ok response throws and lands in the catch block -- same silent-fail behavior, just via different code path.

**For useSessionTokenUsage:** The current code chain is `fetch().then(res => res.ok ? res.json() : null).then(data => ...)`. With apiFetch, it becomes `apiFetch<T>().then(data => ...).catch(() => {})` because apiFetch throws on non-ok (replacing the `res.ok ?` check with catch).
