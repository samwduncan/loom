**Part A: Buffer flush interval — every animation frame (~16ms)**

The current rAF-gated buffer in `useChatRealtimeHandlers.ts` already uses `requestAnimationFrame`, which fires once per animation frame (~16ms). The current pattern accumulates text between frames and flushes on the next rAF callback, which is correct behavior for per-frame flush.

However, there is a subtle issue: the current pattern only schedules ONE rAF per incoming chunk burst. If multiple WebSocket messages arrive within a single frame, they all accumulate in `streamBufferRef` and flush together on the next frame — this is the desired behavior (per-frame flush, not per-message flush).

Verify the current pattern matches the user decision by confirming:
1. `requestAnimationFrame` (not `setTimeout`) is used for flush scheduling — already true
2. No additional debounce or throttle wraps the rAF call — already true
3. The buffer flushes exactly once per frame, batching all chunks received between frames — already true

**Part B: Hybrid markdown rendering**

In `Markdown.tsx`, implement hybrid markdown rendering that renders inline markdown (bold, italic, inline code) live during streaming but defers block elements (code blocks, headers, lists) until their delimiter completes.

The approach: When `isStreaming` is true, preprocess the `children` string before passing it to ReactMarkdown. The preprocessing function `preprocessStreamingMarkdown(text: string)` should:

1. **Detect incomplete block elements at the end of the string:**
   - Incomplete code fence: String ends with ``` followed by content but no closing ``` — handled by Part C (code block container)
   - Incomplete header: String ends with `\n#` or `\n##` etc. without a newline after — strip the incomplete header line
   - Incomplete list item: String ends with `\n- ` or `\n* ` or `\n1. ` without content — strip the incomplete list marker

2. **Leave inline markdown untouched:** Bold (`**text**`), italic (`*text*`), inline code (`` `text` ``), links, etc. render live because ReactMarkdown handles these correctly even mid-stream.

3. **Return the preprocessed string** to pass to ReactMarkdown.

Create a helper function `preprocessStreamingMarkdown` inside Markdown.tsx (not exported — internal utility):

```typescript
function preprocessStreamingMarkdown(text: string): string {
  // Don't process if empty
  if (!text) return text;

  // Find the last newline
  const lastNewlineIdx = text.lastIndexOf('\n');
  if (lastNewlineIdx === -1) return text; // Single line — render as-is

  const lastLine = text.slice(lastNewlineIdx + 1);

  // Incomplete header (line starts with # but no content or still typing)
  if (/^#{1,6}\s*$/.test(lastLine)) {
    return text.slice(0, lastNewlineIdx);
  }

  // Incomplete list item (line starts with marker but no content)
  if (/^(\s*[-*+]|\s*\d+\.)\s*$/.test(lastLine)) {
    return text.slice(0, lastNewlineIdx);
  }

  return text;
}
```

In the Markdown component, when `isStreaming` is true, preprocess:
```tsx
const processedContent = isStreaming ? preprocessStreamingMarkdown(children) : children;
// Pass processedContent to ReactMarkdown instead of children
```

**Part C: Immediate code block container on ``` detection**

In `Markdown.tsx`, add a custom `code` component override that, during streaming, immediately creates the code block container when opening ``` is detected. ReactMarkdown with `remarkGfm` already parses fenced code blocks, but during streaming an unclosed fence would be rendered as inline text.

The approach: In the `preprocessStreamingMarkdown` function, detect an unclosed code fence:

```typescript
// Count ``` occurrences (triple backtick fences)
const fenceMatches = text.match(/^```/gm);
const fenceCount = fenceMatches ? fenceMatches.length : 0;

// If odd number of fences, we have an unclosed code block
if (fenceCount % 2 === 1) {
  // Find the last opening fence
  const lastFenceIdx = text.lastIndexOf('\n```');
  const fenceStart = lastFenceIdx === -1 ? text.indexOf('```') : lastFenceIdx + 1;

  // Extract the fence line (may have language specifier)
  const fenceLineEnd = text.indexOf('\n', fenceStart);
  const fenceLine = fenceLineEnd === -1 ? text.slice(fenceStart) : text.slice(fenceStart, fenceLineEnd);
  const codeContent = fenceLineEnd === -1 ? '' : text.slice(fenceLineEnd + 1);

  // Close the fence so ReactMarkdown creates a proper code block
  return text + '\n```';
}
```

This appends a synthetic closing fence to the preprocessed string, so ReactMarkdown renders a proper `<pre><code>` block immediately. As more code streams in, the content inside the block grows. When the real closing ``` arrives, the synthetic one is no longer needed (fence count becomes even).

Place the unclosed-fence check BEFORE the incomplete header/list checks in `preprocessStreamingMarkdown`, since it modifies the full string.
