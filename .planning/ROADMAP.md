### v3.1 "The App (Rebuilt)" (Phases 74-77) -- IN PROGRESS

**Milestone Goal:** Build a fresh iOS chat app using Private Mind as read-only reference -- clone ChatGPT/Claude iOS quality for drawer navigation, session management, and chat UI, then layer on Loom's unique tool call visualization and permission system. Keep all shared business logic (stores, WebSocket, multiplexer, auth). Rebuild all UI components and route screens from scratch.

**Testing Policy:** Every phase includes tests as a success criterion. Tests are written alongside implementation, not deferred. Coverage targets: store logic, state machines, data transformations, component rendering, and integration flows. Tests catch regressions early and serve as living documentation of expected behavior.

- [ ] **Phase 74: Shell & Connection** - Auth, WebSocket lifecycle, drawer navigation, theme system, safe areas
- [ ] **Phase 75: Chat Core** - Message list (FlatList inverted), streaming markdown, composer with keyboard avoidance, thinking indicator
- [ ] **Phase 76: Session & Interaction** - Session management (grouping, search, pin, delete, context menus), scroll-to-bottom, copy, prompt suggestions
- [ ] **Phase 77: Loom Integration** - Tool call cards, permission approve/deny, accordion grouping, spring animations, glass surfaces

## Phase Details

### Phase 74: Shell & Connection
**Goal**: User can launch the app, authenticate, and navigate a working drawer/chat shell with live WebSocket connection -- the app stays connected, reconnects on disruption, and respects iOS safe areas on all screens
**Depends on**: Nothing (first phase of v3.1; inherits shared/ and Expo scaffold from v3.0)
**Requirements**: CONN-01, CONN-02, CONN-03, CONN-04, CONN-05, CONN-07, NAV-01, NAV-02, NAV-03, NAV-04
**Success Criteria** (what must be TRUE):
  1. User sees a login screen, enters credentials, and lands on the chat screen with auth token persisted in iOS Keychain -- closing and reopening the app skips login
  2. WebSocket connects on app launch, reconnects with exponential backoff on disconnect, and a connection banner appears when the server is unreachable
  3. User can swipe right or tap the hamburger to open a slide drawer, and the main content shifts with parallax effect during the transition
  4. All screens respect iOS safe areas -- no content hidden behind the Dynamic Island notch or home indicator
  5. App resumes WebSocket connection on foreground return with correct auth (no stale token, no duplicate connections)
  6. Tests cover auth flow (login, token persist, token clear), WebSocket lifecycle (connect, reconnect, foreground resume), and drawer navigation state
**Plans**: TBD
**UI hint**: yes

### Phase 75: Chat Core
**Goal**: User can send a message and see a streaming markdown response with syntax highlighting, thinking indicators, and smooth keyboard handling -- the core chat loop feels as polished as ChatGPT iOS
**Depends on**: Phase 74 (auth + WebSocket + navigation shell)
**Requirements**: CHAT-01, CHAT-02, CHAT-03, CHAT-04, CHAT-09, COMP-01, COMP-02, COMP-03, COMP-06
**Success Criteria** (what must be TRUE):
  1. Messages render in an inverted FlatList -- user bubbles right-aligned with accent color, assistant messages left-aligned with neutral background
  2. Streaming markdown renders with syntax highlighting as the AI responds, and the message list auto-scrolls to follow new content
  3. A thinking indicator appears before the first token arrives, and thinking disclosure blocks expand/collapse after completion
  4. Composer auto-grows as the user types (up to max height then scrolls), keyboard pushes content up smoothly via react-native-keyboard-controller with no jumps
  5. Send button reflects state (disabled when empty, accent when ready, stop icon during streaming) and tapping stop halts the active stream
  6. Tests cover message rendering (bubble alignment, role distinction), streaming state transitions, composer state machine, and keyboard avoidance layout
**Plans**: TBD
**UI hint**: yes

### Phase 76: Session & Interaction
**Goal**: User can manage their conversation history and interact with messages beyond basic reading -- sessions are organized, searchable, and manageable through native iOS patterns (context menus, swipe actions), and the chat surface has scroll-to-bottom, copy, and prompt suggestions
**Depends on**: Phase 75 (chat UI rendering messages)
**Requirements**: SESS-01, SESS-02, SESS-03, SESS-04, SESS-05, SESS-06, SESS-07, SESS-08, CHAT-05, CHAT-06, COMP-04, COMP-05
**Success Criteria** (what must be TRUE):
  1. Sessions in the drawer are grouped by date (Today, Yesterday, Last Week, etc.) and by project with collapsible project headers
  2. User can create a new session, switch sessions by tapping, search sessions by title, and pin sessions to the top of the list
  3. User can delete a session via swipe or native iOS context menu (zeego), and the context menu also offers rename and pin options
  4. Scroll-to-bottom button appears when the user scrolls up in the message list, and tapping it returns to the latest message
  5. User can copy message text via long-press, receives haptic feedback on message send, and sees prompt suggestion chips on a new empty chat
  6. Tests cover session CRUD operations, date/project grouping logic, search filtering, pin persistence, and context menu actions
**Plans**: TBD
**UI hint**: yes

### Phase 77: Loom Integration
**Goal**: Loom's unique power -- tool call visualization and permission management -- renders inline in the chat stream with the same quality as the rest of the app, elevated by spring animations, glass surfaces, and the Soul doc visual language
**Depends on**: Phase 75 (message rendering), Phase 76 (interaction patterns established)
**Requirements**: TOOL-01, TOOL-02, TOOL-03, TOOL-04, TOOL-05, TOOL-06, CHAT-07, CHAT-08, CONN-06
**Success Criteria** (what must be TRUE):
  1. Tool call cards (Bash, Read, Edit, Write, Glob, Grep) appear inline in the message stream with state transitions (pending, running, complete/error) and expand/collapse for details
  2. Permission requests appear as elevated inline cards with single-tap approve/deny and haptic feedback on action
  3. Multiple consecutive tool calls collapse into an accordion group that can be expanded individually
  4. Messages appear with spring-physics entering animations, and elevated surfaces (permission cards, tool groups) use glass/blur effects per Soul doc hierarchy
  5. Stream content persists during app background via MMKV snapshot -- returning to the app shows where the stream left off without data loss
  6. Tests cover tool card state machine (pending/running/complete/error), permission approve/deny flow, accordion grouping logic, and MMKV background persistence

## Known Risks & Contingencies

| Risk | Impact | Mitigation |
|------|--------|------------|
| FlatList inverted scroll performance | Scroll jank at 50+ messages | React.memo on message components, estimatedItemSize tuning; FlashList v2 as drop-in fallback |
| react-native-enriched-markdown streaming edge cases | Rendering glitches during fast streaming | ChunkedCodeView pattern (swift-chat) -- freeze completed blocks, only update trailing |
| Keyboard controller integration complexity | Layout jumps, inconsistent behavior | Follow Private Mind's exact KeyboardAvoidingView wrapper pattern |
| Apple Developer enrollment still pending | Blocks device testing | Check enrollment status at Phase 74 start; proceed with simulator if needed |
| Phase 69 code quality unknown (being replaced) | Shared hooks/lib may have latent bugs | Keep shared/ and hooks/ but verify each import works correctly during Phase 74 |
| Glass/blur effects performance on older devices | Frame drops during animations | Performance-gate blur: disable on < A15 chip, use solid backgrounds as fallback |
