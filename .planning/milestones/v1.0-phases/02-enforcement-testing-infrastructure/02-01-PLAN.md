Create `src/eslint-rules/` directory with 9 custom ESLint rule files (plain .js with JSDoc annotations, NOT .ts -- ESLint loads rules at config time before any build step). All files use ESM (`export default`).

**Rule specifications per CONTEXT.md locked decisions:**

1. **no-hardcoded-colors.js** -- Constitution 3.1
   - Ban Tailwind color utilities with color name + number in className/cn() args: regex `\b(?:bg|text|border|ring|outline|shadow|divide|from|via|to|decoration|accent|caret|fill|stroke)-(?:slate|gray|zinc|neutral|stone|red|orange|amber|yellow|lime|green|emerald|teal|cyan|sky|blue|indigo|violet|purple|fuchsia|pink|rose|black|white)-\d{2,4}\b`
   - Ban hardcoded hex in className arbitrary values: `(?:bg|text|border|ring|outline)-\[#[0-9a-fA-F]{3,8}\]`
   - Check: `JSXAttribute[name.name="className"] Literal`, `TemplateLiteral TemplateElement`, and `CallExpression[callee.name="cn"]` arguments
   - **ADDITIONALLY (Issue #2 fix -- shallow color detection):** Add a SEPARATE AST visitor that checks ALL JSX attribute string literal values (not just className/cn()) for raw hex color patterns matching `#[0-9a-fA-F]{3,8}` (standalone hex, NOT inside Tailwind arbitrary value syntax). This catches `<Icon color="#ff0000" />`, `<div fill="#abc" />`, etc. Implementation:
     - Visit ALL `JSXAttribute` nodes where the value is a `Literal` (string)
     - Skip attributes named `className` (already covered by the Tailwind-specific checks above)
     - Test the string value against `/^#[0-9a-fA-F]{3,8}$/` (the ENTIRE value is a hex color) OR `/(?:^|[\s,;])#[0-9a-fA-F]{3,8}(?:$|[\s,;])/` (hex embedded in a larger string)
     - Report with a new messageId: `hexInAttribute: 'Hardcoded hex color "{{match}}" in JSX attribute is banned. Use a CSS custom property or semantic token.'`
     - Do NOT check non-string values (expressions like `{someVar}` are fine -- they're dynamic)
   - **ADDITIONALLY (Issue #5 fix -- arbitrary value catch):** The `HEX_IN_CLASSNAME_PATTERN` regex already covers `bg-[#...]`, `text-[#...]` etc. Explicitly add ALL utility prefixes that support arbitrary color values to the regex: `(?:bg|text|border|ring|outline|shadow|divide|from|via|to|decoration|accent|caret|fill|stroke)-\[#[0-9a-fA-F]{3,8}\]`. This catches arbitrary hex values across ALL Tailwind color utilities, not just the 5 originally listed. Keep the named-color-scale blacklist as-is (it covers all standard Tailwind scales). This DUAL approach (blacklist for named scales + catch for arbitrary hex values) provides comprehensive coverage without needing a brittle whitelist.

2. **no-raw-z-index.js** -- Constitution 3.3
   - Ban `z-[<number>]` arbitrary z-index: regex `\bz-\[\d+\]`
   - Ban Tailwind utility z-index `z-0` through `z-50` etc: regex `\bz-\d+\b` (except allow `z-auto`)
   - Check same className/cn() locations as no-hardcoded-colors

3. **no-classname-concat.js** -- Constitution 3.6
   - Ban `BinaryExpression[operator="+"]` directly inside `JSXAttribute[name.name="className"] > JSXExpressionContainer`
   - Ban `TemplateLiteral` directly inside `JSXAttribute[name.name="className"] > JSXExpressionContainer` (raw template literals, NOT inside cn() calls)
   - Allow: `cn()` call expressions in className (that IS the required pattern)

4. **no-whole-store-subscription.js** -- Constitution 4.2
   - Match `CallExpression` where callee is `useTimelineStore`, `useStreamStore`, `useUIStore`, or `useConnectionStore` with 0 arguments
   - Error message: "Use a selector: useXStore(state => state.field)"

5. **no-external-store-mutation.js** -- Constitution 4.5
   - Match `MemberExpression` where object is one of the four store identifiers and property is `setState` or `getState`
   - Only flag in component files (.tsx), NOT in store definition files
   - Use filename check: flag if filename ends with `.tsx` or does NOT include `/stores/`

6. **no-banned-inline-style.js** -- Constitution 3.2 + CONTEXT.md allowlist
   - Flag `JSXAttribute[name.name="style"]` with `JSXExpressionContainer` containing `ObjectExpression`
   - **Two-tier validation (Issue #1 fix -- style value loophole):**
     - **Tier 1 -- Property name allowlist:** Allowlisted property names: `width`, `height`, `minWidth`, `minHeight`, `maxWidth`, `maxHeight`, `transform`, `translate`, `opacity`, `clipPath`, `top`, `left`, `right`, `bottom`, `inset`, `position`, `zIndex`, `flex`, `flexGrow`, `flexShrink`, `flexBasis`, `gridColumn`, `gridRow`, `gap`, `overflow`
     - Any property NOT in the allowlist triggers error: "Inline style property '{{prop}}' is banned. Use Tailwind utilities or CSS custom properties."
     - **Tier 2 -- Token-backed value validation:** For properties that have corresponding design token categories, the VALUE must be a `var(--...)` reference, NOT a raw number or string. Token-backed properties requiring var() values: `zIndex`, `gap`. Implementation: when the property name is in the token-backed set, check the value node. If the value is a `Literal` (number or string not starting with `var(`), report: "Inline style '{{prop}}' must use a token variable (e.g., `var(--z-modal)`), not a raw value."
     - Properties that MAY use raw values (dynamic layout): `width`, `height`, `minWidth`, `minHeight`, `maxWidth`, `maxHeight`, `transform`, `translate`, `opacity`, `clipPath`, `top`, `left`, `right`, `bottom`, `inset`, `position`, `flex`, `flexGrow`, `flexShrink`, `flexBasis`, `gridColumn`, `gridRow`, `overflow`. These are excluded from Tier 2 checking.

7. **no-any-without-reason.js** -- Constitution 5.2 + CONTEXT.md exception pattern
   - Visit `TSAnyKeyword` nodes
   - Check for trailing comment on the same line matching `// ANY: ` followed by 10+ non-placeholder characters
   - Reject reasons containing only: "TODO", "fix later", "temp", "temporary", "fixme", "hack" (case-insensitive substring match for these forbidden words)
   - Error: "Explicit `any` type requires `// ANY: [reason]` comment with 10+ character reason"

8. **no-non-null-without-reason.js** -- Constitution 5.2 + CONTEXT.md exception pattern
   - Visit `TSNonNullExpression` nodes
   - Check for trailing comment on same line matching `// ASSERT: ` followed by 10+ chars
   - Same placeholder rejection as no-any-without-reason
   - Error: "Non-null assertion requires `// ASSERT: [reason]` comment with 10+ character reason"

9. **no-token-shadowing.js** -- CONTEXT.md decision
   - This rule only applies to CSS files, but ESLint can still check for it if we parse string content. HOWEVER, since ESLint doesn't parse CSS natively, implement this as a rule that checks template literals and string literals containing CSS custom property declarations with banned prefixes: `--surface-`, `--accent-`, `--text-`, `--border-`, `--z-`, `--space-`, `--font-`, `--ease-`, `--duration-`, `--fx-`.
   - In practice, flag any JSX `style` prop or CSS-in-JS string that declares (assigns to) a CSS custom property with these prefixes.
   - This is a best-effort guard -- the real enforcement is that tokens.css is the single source of truth.
