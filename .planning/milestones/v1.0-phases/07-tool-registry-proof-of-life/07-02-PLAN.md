  <name>Task 1: ActiveMessage multi-span refactor + useStreamBuffer ref-switching + double-init guard</name>
  <files>src/src/hooks/useStreamBuffer.ts, src/src/hooks/useStreamBuffer.test.ts, src/src/components/chat/view/ActiveMessage.tsx, src/src/components/chat/view/ActiveMessage.test.tsx, src/src/lib/websocket-init.ts</files>
  <action>
    This is the most technically demanding task. The core change: ActiveMessage goes from a single text span to a segment array (text spans interleaved with ToolChip components). The rAF loop paints to the current active text span.

    **1. useStreamBuffer extension** — Minimal change. The hook already writes `bufferRef.current += token` and paints to `textNodeRef.current.textContent`. The key change: `textNodeRef` is now a mutable ref that the parent (ActiveMessage) can swap to point at different span DOM nodes. When the ref is null during a span switch, tokens accumulate in `bufferRef` harmlessly and paint on the next frame when the new ref is available.

    The critical concern from research: when switching spans, the buffer contains ALL accumulated text. The new span should only show NEW text. Solution: Add a `bufferOffsetRef` that tracks how many characters have been "committed" to previous spans. The paint function writes `node.textContent = bufferRef.current.slice(bufferOffsetRef.current)` instead of the full buffer. When ActiveMessage switches to a new span, it calls a new `checkpoint()` method that sets `bufferOffsetRef.current = bufferRef.current.length`.

    Update `UseStreamBufferReturn` to include `checkpoint: () => void` -- called by ActiveMessage when it switches text span refs.

    Keep `getText()` returning the full buffer (needed for flush).

    Update existing tests to cover the new checkpoint behavior. Add tests:
    - After checkpoint(), new text only paints characters received after the checkpoint
    - Multiple checkpoints work (simulating multiple tool call insertions)

    **2. ActiveMessage multi-span refactor:**

    Replace the single `<span ref={textRef}>` with a segment-based architecture:

    - Type: `type Segment = { type: 'text'; id: string } | { type: 'tool'; toolCallId: string }`
    - State: `const [segments, setSegments] = useState<Segment[]>([{ type: 'text', id: initialId }])`
    - `currentTextRef = useRef<HTMLSpanElement | null>(null)` -- always points to the latest text span
    - Subscribe to `useStreamStore(s => s.activeToolCalls.length)` -- detect new tool calls by length change
    - Keep a `knownToolCountRef = useRef(0)` to track how many tool calls are already in the segment list
    - When `activeToolCalls.length > knownToolCountRef.current`:
      1. Read the new tool calls from the store (slice from knownToolCountRef.current)
      2. Call `checkpoint()` on useStreamBuffer (saves current buffer position)
      3. Set `currentTextRef.current = null` (signal ref is switching)
      4. Append new segments: for each new tool call, add `{ type: 'tool', toolCallId: tc.id }` then `{ type: 'text', id: newId }`
      5. Update `knownToolCountRef.current`
    - Each text segment renders a `<span>` with a ref callback. The LAST text span's ref callback sets `currentTextRef.current` to the DOM node.
    - ToolChip segments look up the tool call from the store by ID.

    **ThinkingDisclosure integration:** Above the segments, render `<ThinkingDisclosure thinkingState={thinkingState} />` where `thinkingState = useStreamStore(s => s.thinkingState)`.

    **Finalization:** `handleFlush` still works -- it uses `getText()` which returns the full buffer. Text-only flush to timeline store (per research recommendation: M1 finalized messages contain text only, tool calls visible during streaming but not persisted on finalization).

    **Re-render scope:** Container-level re-renders happen when segments array changes (new tool call). This is expected and acceptable. Text updates remain zero re-render via rAF.

    Update existing ActiveMessage tests to verify:
    - Still streams text with zero re-renders on text updates
    - Still flushes to timeline on finalization
    - ThinkingDisclosure renders when thinkingState is present

    **3. websocket-init.ts double-init guard:**
    Add a module-scoped `let isInitialized = false` flag. At the top of `initializeWebSocket()`, check `if (isInitialized) return`. Set `isInitialized = true` before doing anything. This prevents double-connection if React strict mode double-mounts or if the user navigates away and back.

    **ESLint considerations:**
    - ActiveMessage already has ESLint overrides for store access patterns -- maintain those
    - `gridTemplateRows` is in the allowlist (Phase 3 decision)
    - Test files exempt from no-external-store-mutation (Phase 3 decision)
    - `currentTextRef.current = null` assignment must be in an effect or callback (React 19 react-hooks/refs rule)
  </action>
