  <action>
    Modify ActiveMessage to group consecutive tool segments into ToolCallGroup during streaming.

    **Approach:** Instead of modifying the segment array structure (which would require complex retroactive wrapping), use a simpler rendering-time approach:

    1. Keep the existing segment array as-is (tool + text segments appended normally).

    2. Add a new rendering step: when mapping segments to JSX, detect consecutive tool segments (where interstitial text spans are empty) and group them.

    3. Implementation details:

    a) Add a `Segment` type variant: keep existing types, but add a `| { type: 'group'; toolCallIds: string[] }` variant for the rendered output.

    b) Create a `groupSegments` function that takes the segments array and produces a "rendered segments" array:
       - Scan segments linearly
       - When 2+ consecutive tool segments appear (with only empty text spans between them), replace them with a single group segment
       - Text spans between grouped tools are absorbed (they're empty anyway -- the buffer only writes to the LAST text span)
       - The last text span after a group is kept (it receives new streaming content)

    c) BUT: this grouping must NOT run during the render-time "adjust state during rendering" block -- it should run AFTER segments state is established. The grouping is a derived computation from segments + tool call data.

    d) Simpler approach (recommended): DON'T modify the segment model. Instead, modify the JSX rendering:
       - After segments are mapped, in the render return, use a `useMemo` that processes segments into "render groups"
       - A render group is either a single text span, a single tool chip, or a ToolCallGroup wrapping 2+ consecutive tool IDs
       - The grouping logic checks: for each tool segment, is the next segment also a tool (or a text-then-tool where the text span has no content)?

    e) Actually, the simplest correct approach: check the text span DOM content at render time. Text spans between tools are refs -- we can't easily check their content in the render body.

    f) **Final approach (cleanest):** Track "consecutive tool runs" as a derived state from the segments array structure. Two tool segments are consecutive if the only segment between them is a text segment with no ID that has received content. Since we can't check DOM content in render, use a different signal: the buffer only writes to the LAST text span. So any text span that is NOT the last one is guaranteed empty (it was checkpointed when the next tool arrived). Therefore: all tool segments except possibly the last one in the array are part of a consecutive run if there are no non-trailing text segments with content between them.

    g) **Practical implementation:**
       - In the JSX render body, build "render chunks" from segments:
       - Walk segments. Accumulate consecutive tool segments into a run. A text segment breaks the run ONLY if it's the last segment (it's receiving new content) or if it's followed by another text segment (not a tool).
       - Actually even simpler: in historical messages, ALL tools are at the end. In streaming, tools arrive with empty text spans between them. The pattern is always: `[text, tool, text(empty), tool, text(empty), tool, text(active)]`. So ALL tool segments except the text span after the last tool are part of one consecutive group.
       - Group ALL tool segments into one ToolCallGroup. The trailing text span renders after the group.

    h) **Implementation:**
       - Import `ToolCallGroup` and `ToolCallGroupFromStore` (a new local component similar to `ToolChipFromStore`)
       - Create `ToolCallGroupFromStore` component: subscribes to individual tool calls from stream store, passes them to `ToolCallGroup` with `defaultExpanded={true}` (streaming groups start expanded)
       - In the render, separate segments into: leading text segments (before first tool), tool segments (all consecutive tools), trailing text segment (after last tool)
       - Render: leading text spans + ToolCallGroupFromStore(if 2+ tools, else individual ToolChipFromStore) + trailing text span + cursor
       - This works because streaming tools are ALWAYS consecutive (no real text content between them)

    4. Create `ToolCallGroupFromStore` as a local component in ActiveMessage:
       ```typescript
       function ToolCallGroupFromStore({ toolCallIds }: { toolCallIds: string[] }) {
         // Subscribe to each tool call individually for granular updates
         const toolCalls = useStreamStore((state) =>
           toolCallIds.map(id => state.activeToolCalls.find(tc => tc.id === id)).filter(Boolean)
         );
         if (toolCalls.length < 2) {
           // Demoted to singles (shouldn't happen, but safety)
           return toolCalls.map(tc => <ToolChip key={tc.id} toolCall={tc} />);
         }
         const errors = toolCalls.filter(tc => tc.isError);
         const nonErrors = toolCalls.filter(tc => !tc.isError);
         // During streaming, show group expanded so user sees what's happening
         return <ToolCallGroup tools={nonErrors} errors={errors} defaultExpanded={true} />;
       }
       ```

    5. The key insight for avoiding flicker (Pitfall 1): the segment array is never retroactively modified. Individual ToolChipFromStore is rendered until the 2nd tool arrives, at which point the ENTIRE render output changes in a single React commit. Since the "adjust state during rendering" pattern triggers a synchronous re-render, the user sees the group container appear atomically.

    6. Handle the edge case: if there's only 1 tool segment, render it as `ToolChipFromStore` (no grouping). Once the 2nd tool segment appears, the render body detects 2+ consecutive tools and switches to `ToolCallGroupFromStore`. This switch happens in a single render commit via the "adjust state during rendering" pattern.

    IMPORTANT: Do NOT modify the existing segment model, checkpoint logic, or text ref management. Only change the JSX rendering to group consecutive tool segments.
