       - Define shared test fixtures as an array of `{ name: string, input: string, description: string }`:
         - "basic formatting": `**bold** and *italic* and \`code\``
         - "headings": `# H1\n## H2\n### H3`
         - "code fence": "```js\nconst x = 1;\n```"
         - "unclosed code fence": "```js\nconst x = 1;"
         - "nested lists": `- item 1\n  - nested\n- item 2`
         - "blockquote": `> quoted text\n> more quoted`
         - "incomplete table": `| col1 | col2 |\n| --- | --- |\n| a`
         - "mixed content": A realistic 500-char assistant response with headings, code, lists, bold
         - "large content": A 5KB realistic response to test performance
         - "xss attempt": `<script>alert(1)</script><img onerror="x" src=y>`
       - Export `runCustomConverter(input: string): string` -- wraps convertStreamingMarkdown
       - Export `runStreamdownConverter(input: string): string` -- wraps Streamdown's API (adapt to whatever API Streamdown exposes; if it's a React component rather than a function, document this as an integration concern)
       - Export `evaluateFixture(fixture): { custom: string, streamdown: string, notes: string }` -- runs both converters on same input, captures output

    3. Create `src/src/lib/streamdown-eval.test.ts`:
       - Import fixtures and evaluation functions
       - For each fixture, run both converters and assert:
         a. Neither throws an error
         b. Both produce non-empty output (unless input is empty)
         c. XSS fixture: neither output contains `<script>` or `onerror`
       - Add descriptive test names documenting findings (these tests serve as the evaluation record)
       - Add a summary test that logs the comparison matrix to console (fixture name, custom output length, streamdown output length, visual quality notes)

    4. Evaluate on four criteria (document findings in test file comments and SUMMARY.md):
       - **Visual quality**: Which output looks more like properly formatted markdown? (Both outputs as HTML strings -- compare structure)
       - **Edge cases**: Which handles unclosed fences, partial tables, mixed formatting better?
       - **Integration fit**: Does Streamdown work as a pure function callable from rAF, or does it require its own React render cycle? (Critical: our architecture needs `string -> string` that runs in rAF)
       - **Bundle size**: `cd /home/swd/loom/src && du -sh node_modules/streamdown/` vs custom converter (single file)

    5. Based on evaluation:
       - If custom converter wins: `cd /home/swd/loom/src && npm uninstall streamdown` and remove eval files (or keep as documentation)
       - If Streamdown wins: replace convertStreamingMarkdown with Streamdown wrapper, update useStreamBuffer import
       - Document decision rationale

    NOTE: If Streamdown's API is purely a React component (not a pure function), that's an automatic disqualification for our rAF architecture. Document this finding and skip the full comparison -- our custom converter wins by integration fit alone.
