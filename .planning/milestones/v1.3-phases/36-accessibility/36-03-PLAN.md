    **Global reduced-motion override in base.css:**
    Add at the END of `src/src/styles/base.css` (after all @font-face declarations and base resets):

    ```css
    /* ─── Reduced Motion: Global Override (A11Y-05) ────────────────────
     * Disables ALL animations and transitions when user prefers reduced motion.
     * Uses 0.01ms (not 0ms) to preserve animationend/transitionend event firing
     * for JS logic that depends on those events (e.g., ActiveMessage crossfade).
     */
    @media (prefers-reduced-motion: reduce) {
      *,
      *::before,
      *::after {
        animation-duration: 0.01ms !important;
        animation-iteration-count: 1 !important;
        transition-duration: 0.01ms !important;
        scroll-behavior: auto !important;
      }
    }
    ```

    This single rule covers:
    - All ~30 uses of Tailwind `animate-pulse` and `animate-spin`
    - All custom @keyframes (skeleton-pulse, spin, shimmer, session-pulse, cursor-pulse, thinking-pulse, tool-chip-pulse, permission-slide-in/out, countdown-pulse, composer-spin, shiny-sweep, electric-top/bottom)
    - All `transition:` properties across all CSS files
    - tw-animate-css dialog animations (animate-in, animate-out, fade-in, zoom-in)
    - The `!important` ensures no specificity conflicts

    **Remove redundant per-file reduced-motion blocks:**
    Several CSS files already have per-component `@media (prefers-reduced-motion: reduce)` blocks. These are now redundant since the global rule covers everything. KEEP THEM — they're harmless and provide documentation of intent. Do NOT remove them (removing adds risk for zero benefit).

    **motion.ts — prefersReducedMotion helper:**
    Add to `src/src/lib/motion.ts`:
    ```typescript
    /** Check if user prefers reduced motion. Use for JS-driven animations (framer-motion). */
    export function prefersReducedMotion(): boolean {
      return window.matchMedia('(prefers-reduced-motion: reduce)').matches;
    }
    ```
    This is for any future framer-motion usage — LazyMotion is configured in tokens.css but barely used currently. The global CSS override handles CSS animations; this helper is for imperative JS animations.

Verify no `animation-duration` override in these keyframes that would fight the global rule. If any `!important` durations exist on animation properties in these files, they'd conflict — check and remove if found.
    - `composer.css` has `@keyframes composer-spin` with no reduced-motion block. Same check.

    **Contrast audit test (a11y-contrast.test.ts):**
    Create `src/src/tests/a11y-contrast.test.ts` that mathematically verifies WCAG AA contrast ratios for all text/surface token pairs used in the app.

    The test file should:

    1. **Define OKLCH-to-sRGB conversion functions** inline (no external dependency needed):
       - `oklchToLinearRGB(L, C, H)` — convert OKLCH to linear sRGB via OKLab intermediate
       - `linearToSRGB(c)` — apply gamma correction (c <= 0.0031308 ? 12.92*c : 1.055*c^(1/2.4) - 0.055)
       - `srgbRelativeLuminance(r, g, b)` — 0.2126*r + 0.7152*g + 0.0722*b (using linear values)
       - `contrastRatio(l1, l2)` — (lighter + 0.05) / (darker + 0.05)

    2. **Parse color tokens from tokens.css:** Read the file, extract all `--name: oklch(L C H)` declarations using regex. Handle both `oklch(L C H)` and `oklch(L C H / alpha)` formats. For alpha < 1 (like borders), composite against the surface they're typically used on (--surface-base for borders).

    3. **Define text-on-surface pairs to check:** These are the actual combinations used in the UI:
       - `--text-primary` on `--surface-base` (main body text) — require 4.5:1
       - `--text-primary` on `--surface-sunken` (sidebar text) — require 4.5:1
       - `--text-primary` on `--surface-raised` (card text) — require 4.5:1
       - `--text-secondary` on `--surface-base` — require 4.5:1
       - `--text-muted` on `--surface-base` — require 4.5:1 (this is the most likely to fail)
       - `--text-muted` on `--surface-sunken` — require 4.5:1
       - `--accent-primary` on `--surface-base` (links, buttons) — require 4.5:1
       - `--accent-primary-fg` on `--accent-primary` (text on accent button) — require 4.5:1
       - `--status-success` on `--surface-base` — require 4.5:1
       - `--status-error` on `--surface-base` — require 4.5:1
       - `--status-warning` on `--surface-base` — require 4.5:1
       - `--status-info` on `--surface-base` — require 4.5:1
       - `--border-interactive` on `--surface-base` (interactive borders) — require 3:1

    4. **Run assertions:** For each pair, compute contrast ratio and assert >= threshold.

    5. **If any pair fails:** The test output should show the actual ratio and the required minimum, making it clear what needs adjusting. If `--text-muted` fails (L:0.60 on L:0.20 — research flagged this as borderline), note the options:
       - Increase `--text-muted` lightness from 0.60 to 0.64-0.65 to pass
       - OR accept it as "large text only" usage if all muted text is >= 18px

    **Token adjustments (if needed):**
    If the contrast test reveals failures, adjust the offending tokens in `tokens.css` with the minimum change needed to pass. Document the change in a CSS comment: `/* Adjusted for WCAG AA: was 0.60, now 0.64 */`. Only adjust if the test fails — don't preemptively change tokens.

    The OKLCH-to-sRGB math is non-trivial but well-documented. Use the standard OKLab intermediary:
    - OKLCH(L,C,H) -> OKLab(L, C*cos(H*pi/180), C*sin(H*pi/180))
    - OKLab -> linear sRGB via the standard 3x3 matrix (LMS intermediate)
    - Clamp linear sRGB to [0,1] before gamma correction

    Reference implementation is available at https://bottosson.github.io/posts/oklab/ — the matrix constants are stable and well-known.
